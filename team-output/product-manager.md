## 一、项目定位与愿景

### 1.1 一句话定位

> **不是"会聊天的旅游攻略"，而是"能看地图、会算路、懂取舍的 AI 旅行规划师"。**

本系统的核心差异在于：AI Agent 具备 **空间感知能力**（通过 WebGIS）和 **自主决策能力**（通过多 Agent 协作），能基于地理空间分析做出推荐，而不仅仅是文本生成。

### 1.2 项目愿景

用户说一句话（如"我明天带父母去杭州玩一天"），系统自动完成：

1. **理解意图** — 解析出"杭州"、"1天"、"家庭游（父母→节奏慢、无障碍）"
2. **空间分析** — 在地图上检索景点、计算距离、分析空间分布
3. **智能规划** — 综合考虑时间约束、交通方式、景点开放时间、人流预测，生成最优路线
4. **地图呈现** — 在 WebGIS 地图上可视化展示完整路线、景点详情、空间关系
5. **交互优化** — 用户可以在地图上拖拽调整，Agent 实时响应并重新优化

## 三、核心功能列表（按优先级排序）

### P0 — 必须有（课设答辩核心演示点）

> 缺少任何一项，答辩无法通过。这是 MVP 的骨架。

| 编号 | 功能模块 | 功能描述 | 答辩演示要点 |
|------|---------|---------|-------------|
| P0-1 | **自然语言意图解析** | 用户用自然语言描述旅行需求，AI Agent 自动提取关键信息：目的地、天数、偏好标签、同行人员、预算范围等 | 演示：输入一段话，系统结构化展示解析结果 |
| P0-2 | **景点空间检索与展示** | 基于目的地在 WebGIS 地图上检索并标注景点，支持按类型/评分/距离筛选 | 演示：地图上展示景点 POI 点，点击查看详情 |
| P0-3 | **智能路线规划** | Agent 综合考虑时间约束、空间距离、景点属性，自动生成多日/单日游览路线 | 演示：输入需求后，地图上绘制出带顺序编号的路线 |
| P0-4 | **路线地图可视化** | 路线在地图上以连线方式展示，包含景点间距离、预计交通时间、游览时长 | 演示：完整的路线地图，有图例、有标注 |

| P0-6 | **对话式交互** | 用户可以追问、修改需求，Agent 保持上下文持续优化方案 | 演示：多轮对话逐步细化方案的过程 |

### P1 — 应该有（体现技术深度）

> 这些功能是拿到 85+ 分数的关键，展示团队对 AI Agent 和 WebGIS 的深度理解。

| 编号 | 功能模块 | 功能描述 | 技术亮点 |
|------|---------|---------|---------|
| P1-1 | **多 Agent 协作架构** | 不同 Agent 负责不同职责（意图解析Agent、空间分析Agent、路线优化Agent、推荐Agent），通过协作框架（如 CrewAI/AutoGen）编排 | 答辩时展示 Agent 之间的消息流转和协作过程 |

## 四、用户故事

### 用户故事 1：自由行旅客的一日游规划

> **作为** 一个计划去杭州自由行的旅客，
> **我希望** 告诉系统"我明天想去杭州玩一天，喜欢自然风景和茶文化，不想太赶"，
> **系统能** 自动在地图上规划一条从早到晚的游览路线，包含西湖、龙井村等景点，标注各段交通方式和时间，
> **以便** 我不需要自己做攻略就能直接按路线游玩。

**验收标准：**
- 输入自然语言后 10 秒内返回路线方案
- 地图上清晰展示 3-5 个景点和它们之间的路线
- 每个景点有游览建议时长和推荐理由
- 路线时间安排合理（考虑开园时间和交通时间）

---

### 用户故事 2：家庭出游的个性化调整

> **作为** 一个带孩子和父母出游的家长，
> **我希望** 告诉系统"我带3岁孩子和70岁老人去厦门玩3天"，
> **系统能** 推荐适合老幼的景点（平坦、有休息设施），每天安排不超过3个景点，并标注附近的亲子/老年友好餐厅，
> **以便** 全家人都玩得舒适不累。

**验收标准：**
- Agent 自动识别"3岁孩子+70岁老人"需要降低行程强度
- 推荐的景点适合老幼（避免高爬山、长距离徒步）
- 每日行程标注总步行距离和预估体力消耗等级
- 可通过对话追加约束（如"第二天下午要回酒店休息"）

---

### 用户故事 3：地图交互与路线微调

> **作为** 一个已经获得初步路线方案的用户，
> **我希望** 在地图上看到推荐路线后，能通过点击/拖拽调整景点顺序或替换某个景点，
> **系统能** 实时重新计算路线并更新时间和距离信息，
> **以便** 我能按自己的偏好微调方案而不需要重新输入所有需求。

**验收标准：**
- 地图上可点击景点进行替换（弹出"换一个"按钮）
- 替换后 Agent 推荐同区域类似景点并保持路线合理性
- 地图路线实时更新
- 调整后的方案保留之前的约束条件

---

### 用户故事 4：多日行程的智能编排

> **作为** 一个计划去成都玩4天的游客，
> **我希望** 系统能把景点按天合理分配，每天的区域相对集中（避免东跑西跑），
> **并** 考虑第一天到达较晚只能玩半天、最后一天要赶飞机等因素，
> **以便** 4天行程不浪费时间在交通上。

**验收标准：**
- 多日行程按天展示，每天有独立路线
- 每天景点空间聚集（利用空间聚类算法）
- 时间安排考虑航班/火车时刻约束
- 总览地图上用不同颜色区分每天的路线

---

### 用户故事 5：空间分析与深度推荐

> **作为** 一个深度旅行爱好者，
> **我希望** 系统不仅推荐热门景点，还能基于空间分析发现"隐藏路线"，
> **比如**"这些景点可以串成一条步行路线，沿途经过3条老街和2个当地人去的早市"，
> **以便** 获得非模板化的独特旅行体验。

**验收标准：**
- Agent 使用空间分析能力（如路径分析、POI密度分析）发现连贯的游览动线
- 推荐结果标注"发现理由"（如"空间连续性好"、"沿途POI丰富"）
- 推荐的非热门景点在地图上可查验证实

---

### 用户故事 6：对话式需求细化

> **作为** 一个不确定自己想去哪的用户，
> **我希望** 跟系统聊天，让它帮我逐步明确需求（"你更喜欢海边还是山城？" "预算大概多少？"），
> **系统能** 像一个旅行顾问一样引导我，最终输出完整方案，
> **以便** 即使我没有明确目标也能获得好的规划。

**验收标准：**
- Agent 主动提问引导用户明确需求（至少 3 轮引导对话）
- 问题有针对性（不是泛泛而问，而是基于已有信息推断下一步）
- 最终输出与对话过程中收集的所有偏好一致

## 五、课设评分关注点与得分策略

### 5.1 老师通常关注的维度

| 评分维度 | 权重估计 | 关注要点 | 我们的应对策略 |
|---------|---------|---------|-------------|
| **技术深度** | ★★★★★ | 不是调 API 就行，要展示对技术的深入理解 | 多 Agent 架构 + GIS 空间分析算法 + 路线优化算法 |
| **系统完整性** | ★★★★☆ | 前端 + 后端 + 数据库 + AI，全链路完整 | 确保每个环节都能跑通，不做半成品 |
| **创新性** | ★★★★☆ | 和别人的课设有什么不同 | AI Agent 自主决策 + WebGIS 空间感知 = 市面没有的组合 |
| **可演示性** | ★★★★☆ | 答辩时能不能直观看到效果 | 地图可视化是最好的演示载体，路线变化一目了然 |
| **工作量** | ★★★☆☆ | 代码量、模块数、技术栈广度 | 功能模块数量 + 文档完整度体现 |
| **文档规范** | ★★★☆☆ | 设计文档、架构图、测试报告 | 团队文档同步产出，答辩前完善 |
| **代码质量** | ★★☆☆☆ | 代码结构、注释、可维护性 | 遵循编码规范，关键模块有单元测试 |

### 5.2 高分策略

#### 策略 1：让 AI Agent "活"起来（最关键）

**❌ 低分做法**：用户输入 → 调用 ChatGPT API → 返回文字 → 结束
**✅ 高分做法**：

```
用户输入 → 意图解析Agent分析需求
         → 空间分析Agent在GIS上检索景点、计算空间关系
         → 路线优化Agent利用算法生成最优路线
         → 可视化Agent将结果渲染到地图
         → 用户可以继续对话修改
```

**关键展示点**：答辩时展示 Agent 之间的消息流转过程（类似 debug 日志），让老师看到"这不是简单的 LLM 调用"。

#### 策略 2：WebGIS 不是装饰，是核心工具

**❌ 低分做法**：地图上只是打几个点
**✅ 高分做法**：

- 空间缓冲区分析：景点周边 X 公里内的餐饮/住宿
- 网络分析：最短路径、服务区分析
- 空间聚类：景点按地理邻近性分组（用于多日行程安排）
- 核密度分析：推荐区域 vs 冷门区域
- 地形分析：步行路线的坡度/海拔变化

#### 策略 3：答辩演示的"高光时刻"设计

1. **开场**：一句话输入 → 地图上实时生成完整路线（视觉冲击）
2. **高潮**：展示 Agent 协作日志（"空间分析Agent发现A和B距离太远，建议调整顺序"）
3. **互动**：老师提出修改意见（"如果下雨呢？"），现场演示约束调整后路线变化
4. **收尾**：展示多日行程的全景地图视图

## 七、差异化亮点（为什么我们不同于携程/马蜂窝）

### 7.1 对比分析

| 维度 | 携程/马蜂窝 | 本系统 |
|------|-----------|-------|
| **规划方式** | 用户手动搜索、筛选、排序 | 自然语言描述，AI 全自动规划 |
| **空间智能** | 展示地图，但不做空间分析 | Agent 主动做空间分析辅助决策 |
| **决策过程** | 黑箱，不知道推荐逻辑 | 透明，Agent 给出每步决策理由 |
| **个性化** | 基于标签筛选，维度有限 | 自然语言理解，支持任意维度组合 |
| **交互模式** | 表单/筛选器（僵化） | 对话式（灵活、可追问、可修改） |
| **路线优化** | 按评分排序，不管空间关系 | 考虑空间连续性的路线编排 |
| **约束处理** | 不支持多约束组合 | 时间+预算+偏好+体力+天气 组合约束 |

### 7.2 三大核心差异化

#### 差异化 1：AI Agent 具备空间感知能力

> 市面上 LLM 旅游助手很多，但**能"看地图"的 AI Agent** 几乎没有。

本系统的 Agent 不只是生成文字，它能：
- 调用 GIS 工具分析景点的空间分布关系
- 计算真实的路径距离和通行时间
- 基于空间聚类将邻近景点安排在同一天
- 做缓冲区分析找到景点周边的配套设施

#### 差异化 2：多 Agent 协作的透明决策

> 不是一个黑箱 LLM，而是一个"旅行规划团队"在为你服务。

系统内部有多个专业 Agent：
- **意图解析 Agent** — 理解你想要什么
- **空间分析 Agent** — 分析地理空间数据
- **路线编排 Agent** — 设计最优游览路线
- **推荐解说 Agent** — 解释为什么这样推荐

答辩时展示 Agent 之间的对话记录，比单一 LLM 调用有说服力得多。

#### 差异化 3：地图即界面

> 不是"聊天窗口里嵌个地图"，而是"地图就是主界面，AI 在地图上工作"。

- 所有推荐、路线、分析结果都直接在地图上可视化
- 用户可以直接在地图上操作（点击、拖拽、框选）
- 空间分析结果以图层方式叠加展示

## 八、风险识别与应对

| 风险 | 影响 | 概率 | 应对策略 |
|------|------|------|---------|
| LLM API 响应慢，影响演示体验 | 高 | 中 | 预计算常见方案 + 流式输出 + 加载动画 |
| 空间分析算法实现复杂 | 高 | 中 | 优先用现成 GIS 库（Turf.js / PostGIS），不手写算法 |
| 景点数据获取困难 | 中 | 低 | 备选方案：使用预置数据集（至少准备一套完整数据） |
| Agent 协作调试困难 | 高 | 高 | 从简单的两 Agent 开始，逐步增加；每个 Agent 独立可测试 |
| 多轮对话上下文丢失 | 中 | 中 | 设计结构化的对话状态管理，不纯依赖 LLM 记忆 |
| 路线可视化效果差 | 中 | 低 | 使用成熟的地图库（Leaflet / Mapbox），参考优秀案例 |
| 时间不够，功能做不完 | 高 | 中 | 严格 MVP 思维，P2 功能提前放弃不纠结 |

## 九、成功标准

### 9.1 MVP 交付标准（答辩通过线）

- [ ] 用户输入自然语言需求，系统返回结构化解析结果
- [ ] 地图上展示目的地景点 POI
- [ ] 系统生成包含 3-5 个景点的一日游路线并在地图上可视化

- [ ] 路线经过优化，对比展示优化效果
- [ ] 支持地图交互（景点替换/顺序调整）

- [ ] 文档完整规范
