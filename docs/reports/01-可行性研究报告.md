# 可行性研究报告

## 基于AI Agent的旅游路线智能规划与景点空间推荐系统

---

## 1 引言

### 1.1 编写目的

本报告旨在回答一个核心问题：**在6周课设周期、2—3人学生团队的约束下，构建一个融合AI Agent与WebGIS的旅游路线规划系统是否可行？如果可行，怎样做最合理？如果存在风险，关键瓶颈在哪里？**

报告不是对既定方案的背书，而是从多个可选方案出发，通过对比分析论证当前方案的合理性，并对进度、成本、风险进行量化评估。

### 1.2 项目背景

旅游路线规划是一个典型的多目标决策问题——用户需综合景点选择、空间距离、时间分配、个人偏好等多重因素。现有方案存在两类不足：

- **传统旅游平台**（携程、马蜂窝）：提供景点列表和攻略，但规划过程依赖用户手动筛选和拼接，不具备空间智能。
- **通用LLM对话工具**（ChatGPT、通义千问）：能理解自然语言并生成旅行建议文本，但缺乏空间分析能力——无法计算景点间真实距离、无法进行空间聚类分天、无法做路径优化。

本项目的核心思路是：**将LLM的意图理解能力与GIS的空间分析能力通过Agent机制结合**，构建一个"能看地图、会做空间分析"的AI旅行规划系统。

### 1.3 定义与缩略语

| 术语 | 定义 |
|------|------|
| AI Agent | 基于LLM的智能代理，具备意图理解、工具调用、多步推理能力 |
| WebGIS | 基于Web的地理信息系统 |
| LLM | 大语言模型（Large Language Model） |
| POI | 兴趣点（Point of Interest） |
| PostGIS | PostgreSQL空间数据库扩展 |
| LangGraph | 基于图结构的LLM Agent编排框架 |
| TSP | 旅行商问题（Traveling Salesman Problem） |
| DBSCAN | 基于密度的空间聚类算法 |
| MMR | 最大边界相关（Maximal Marginal Relevance） |
| SSE | 服务器发送事件（Server-Sent Events） |

## 2 可行性研究的前提

### 2.1 问题定义

本系统需解决的核心问题：**如何让用户用自然语言描述旅行需求，系统自动完成景点筛选、空间分天和路线优化，并将结果在地图上可视化？**

该问题可分解为三个子问题及其难度判断：

| 子问题 | 核心挑战 | 难度 | 说明 |
|--------|----------|------|------|
| 意图解析 | 从非结构化文本中提取目的地、天数、偏好等结构化参数 | 中 | LLM的Function Calling已较成熟，此子问题风险最低 |
| 空间分析与推荐 | 基于地理位置进行POI筛选、聚类分天、路径优化 | 高 | 涉及5个算法环节的串联和调优，是项目核心难点 |
| 结果可视化与交互 | 地图标注、路线绘制、对话与地图联动 | 中 | Leaflet生态成熟，但状态同步逻辑需要设计 |

### 2.2 系统目标

| 编号 | 目标 | 验收标准 |
|------|------|----------|
| G1 | 自然语言意图解析 | 输入"去杭州玩两天，喜欢历史文化"，正确提取目的地、天数、偏好 |
| G2 | 空间POI检索 | 基于目的地和偏好，从PostGIS + 高德API返回相关景点 |
| G3 | 空间感知推荐 | 五步流水线输出合理的多日行程（空间过滤→评分→MMR→DBSCAN→TSP） |
| G4 | 路线规划可视化 | Leaflet地图标注景点、绘制路线，与对话联动 |
| G5 | 多轮对话 | 支持在已有行程基础上修改调整，保持上下文 |
| G6 | 流式响应 | SSE推送LLM输出，首token延迟 < 3秒 |

### 2.3 系统范围

**包含：** 前端对话界面 + 地图可视化 + 后端Agent引擎 + 空间推荐算法 + 数据库 + Docker部署。

**不包含：** 用户注册认证体系、支付与订单系统、UGC社交功能、移动端原生适配。

### 2.4 约束条件

| 约束 | 具体内容 | 影响 |
|------|----------|------|
| 时间 | 6周（30个工作日），扣除需求分析与文档，有效编码约4周 | **关键** |
| 人力 | 2—3名学生，需覆盖前端、后端、算法、测试 | **关键** |
| 技术 | LLM依赖外部API（存在网络延迟、配额限制、响应不确定性） | 中等 |
| 数据 | POI数据依赖高德API + PostGIS本地库 | 低 |

## 3 技术可行性分析

### 3.1 方案比选与论证

在确定最终方案前，需对几个关键技术决策点进行比选。

#### 3.1.1 Agent编排框架选型

| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|----------|
| **LangGraph StateGraph** | 有状态多Agent协作；内置Checkpoint持久化；条件分支灵活 | 学习曲线陡；API迭代快（2024年发布，尚不稳定）；调试复杂 | 多轮对话、需要状态回溯的复杂Agent场景 |
| LangChain简单Chain | 上手快；文档丰富；社区大 | 不支持多Agent状态共享；无法自然实现条件分支 | 单轮问答、简单工具调用 |
| 自研Agent循环 | 完全可控；无框架依赖 | 开发量大（需自实现状态管理、错误处理等）；6周内不现实 | 有充足时间的长期项目 |

**选择LangGraph的理由：** 项目核心需求是"多轮对话 + 多Agent协作（Router→Planner→Formatter）+ 状态持久化"，LangChain简单Chain无法满足。自研循环在6周内不现实。LangGraph虽学习成本高且API不稳定，但其功能与项目需求最为匹配。

**代价：** 团队需投入约2—3天专门学习LangGraph，且开发过程中可能遇到文档与实际行为不一致的问题。这是可接受的代价。

#### 3.1.2 空间数据方案选型

| 方案 | 优势 | 劣势 |
|------|------|------|
| **PostGIS本地库 + 高德API** | 本地空间计算快；减少外部API依赖；支持复杂空间分析 | 需维护本地POI数据；增加部署复杂度 |
| 纯高德API | 零部署成本；POI数据全面 | 每次空间查询都走网络请求；无法做本地聚类分析；日配额限制 |
| SQLite + 手动距离计算 | 轻量部署 | 不支持真正的空间索引和空间函数；Haversine公式无法处理复杂空间操作 |

**选择PostGIS + 高德API的理由：** 项目的核心创新点在于"空间感知推荐"，五步流水线中的空间过滤、DBSCAN聚类、TSP优化都需要本地空间计算能力。纯API方案无法支持本地聚类分析；SQLite缺乏空间索引。PostGIS虽然增加部署复杂度，但通过Docker可一键解决。

#### 3.1.3 前端地图选型

| 方案 | 优势 | 劣势 |
|------|------|------|
| **Vue 3 + Leaflet** | 轻量开源；无需API Key；中文社区好 | 大数据量渲染性能不如WebGL方案 |
| React + MapBox GL | WebGL渲染性能好；地图效果精美 | 需要MapBox API Key；React学习曲线 |
| Vue 3 + 高德JS API | 国内地图数据准确；中文标注完善 | 与后端高德Web API是两套体系；JS API文档质量一般 |

**选择Vue 3 + Leaflet的理由：** 课设场景POI数量有限（通常20—50个），Leaflet性能完全够用。开源免费无需API Key，降低了配置复杂度。与Vue 3的技术栈统一（vs. React）减少了团队学习成本。

### 3.3 关键技术环节可行性逐一论证

#### 3.3.1 LLM工具调用——可行，但需容错设计

通义千问qwen-plus已原生支持Function Calling，可定义工具schema并返回结构化调用参数。LangGraph的ToolNode封装了工具调用的执行和结果回传。

**实际风险：** LLM可能生成错误的工具调用参数（如错误的坐标范围、不合理的POI类型）。根据社区反馈和实测经验，qwen-plus在结构化参数生成上的准确率约85%—90%。

**必要措施：** 参数校验层（坐标范围检查、类型枚举校验）+ 自动重试（最多3次）+ 错误提示反馈给用户。这不是可选的优化，而是必须的基础设施。

#### 3.3.2 五步推荐流水线——项目最大技术风险点

这是项目的核心难点。逐步分析每步的可行性和风险：

| 步骤 | 实现方式 | 单步可行性 | 实际风险 |
|------|----------|-----------|----------|
| ①空间过滤 | PostGIS `ST_DWithin`、`ST_MakeEnvelope` | ✅ 成熟内置函数 | 坐标系转换（WGS84 vs. GCJ02）需正确处理 |
| ②多因子评分 | 加权评分模型 | ✅ 经典方法，实现简单 | 权重初始值设定——可通过Agent动态调整 |
| ③MMR多样性 | `argmax[λ·Rel - (1-λ)·max_sim]` | ✅ 信息检索经典算法 | 计算量与POI数量平方相关，需控制候选集大小 |
| ④DBSCAN聚类分天 | scikit-learn DBSCAN | ⚠️ **参数敏感** | 不同城市POI密度差异大，固定eps效果不稳定 |
| ⑤TSP路线优化 | 最近邻 + 2-opt | ✅ 经典近似算法 | 景点>30个时2-opt可能耗时，需限制迭代次数 |

**关键风险详细分析：DBSCAN参数敏感性。** DBSCAN需要指定eps（邻域半径）和min_samples（最小样本数）。杭州市西湖区POI密度远高于临安区，若eps=1km在西湖区可能将所有POI聚为一类，在临安区则可能每个POI自成一类。

**应对方案（三选一或组合）：**
- 方案A：根据城市POI密度动态计算eps（如取所有POI最近邻距离的中位数）
- 方案B：将eps作为Agent可调参数，由LLM根据城市规模选择
- 方案C：使用OPTICS替代DBSCAN（自动适应不同密度，但实现复杂度更高）

**建议：** 课设场景下采用方案A（动态eps）即可，实现成本低且效果可接受。

**结论：** 五步流水线中每一步单独可行，但**串联后的集成调试和参数调优是项目最大挑战**。建议在第2—3周集中攻克，预留充足时间。

#### 3.3.3 LLM适配层与降级——可行，成本可控

适配器模式是GoF经典设计模式，开发成本主要在接口定义和格式转换。DashScope API兼容OpenAI接口格式，进一步降低了适配工作量。

**降级策略的必要性：** DashScope API存在不可用的可能（服务维护、网络波动、配额超限）。当主LLM不可用时，自动切换至备用LLM（如OpenAI或本地模型）。当所有LLM均不可用时，退化为基于规则的简单推荐（根据目的地直接查询POI并按评分排序），保证系统基本可用。

#### 3.3.4 前端对话-地图联动——可行，设计是关键

Vue 3的响应式系统 + Pinia状态管理天然支持组件间的状态同步。核心设计：对话界面发送请求后更新Store中的`currentPlan`状态，地图组件通过响应式监听自动更新标注和路线。

**性能考量：** 课设场景下POI数量通常20—50个，Leaflet完全胜任。若未来扩展至100+标注点，需引入MarkerCluster插件。

### 3.4 技术成熟度与风险评估

#### 3.4.1 各组件成熟度

| 组件 | 成熟度 | 对课设的影响 |
|------|--------|-------------|
| Vue 3 / Element Plus | ★★★★★ | 零风险，国内生态极好 |
| Leaflet | ★★★★☆ | 低风险，文档完善，社区活跃 |
| FastAPI | ★★★★★ | 零风险，Python Web框架首选 |
| PostgreSQL / PostGIS | ★★★★★ | 零风险，空间数据库事实标准 |
| DashScope | ★★★★☆ | 低风险，国内访问稳定，免费额度充足 |
| GeoAlchemy2 | ★★★★☆ | 低风险，接口与SQLAlchemy一致 |
| **LangGraph** | **★★★☆☆** | **中等风险——2024年发布，API仍在快速迭代，文档可能滞后于实际版本** |

**核心发现：** LangGraph是整个技术栈中成熟度最低的组件。这意味着：
- 团队可能遇到文档与实际API不一致的情况
- 升级版本时可能遭遇Breaking Change
- 遇到Bug时社区解决方案较少

**应对：** 锁定版本号（`langgraph==0.x.x`），开发期间不升级；通过适配层隔离LangGraph特有的API，减少变更影响面。

#### 3.4.2 技术风险量化评估

| 风险 | 概率 | 影响 | 后果描述 | 缓解策略 |
|------|------|------|----------|----------|
| DBSCAN参数不适配特定城市 | 中 | 中 | 某些城市推荐行程分天不合理 | 动态eps计算 + 参数可调配置 |
| LLM工具调用参数错误 | 中 | 高 | 空间查询返回空结果或错误结果 | 参数校验 + 自动重试（≤3次）+ 用户提示 |

| LLM响应超时（>10秒） | 低 | 中 | 用户体验差 | SSE流式推送 + 超时降级 |
| 高德API日配额耗尽 | 低 | 中 | 无法获取实时POI数据 | 本地PostGIS缓存 + 请求去重 |
| TSP优化在大规模POI下超时 | 低 | 低 | 路线优化结果不够优 | 限制TSP输入POI数≤20 |

## 4 经济可行性分析

### 4.1 开发成本分析

#### 4.1.1 人力投入估算

本项目为课程设计，不涉及正式薪资支出，以时间投入衡量：

| 阶段 | 工作内容 | 人天 | 前置依赖 |
|------|----------|------|----------|
| 环境搭建与MVP验证 | Docker环境、LangGraph+DashScope最小闭环、PostGIS初始化 | 3 | 无 |
| 后端核心开发 | Agent引擎、推荐算法流水线（5步）、API接口 | 12 | MVP验证通过 |
| 前端开发 | 对话界面、Leaflet地图组件、联动逻辑 | 10 | 后端API就绪（可用Mock先行） |
| 联调与测试 | 前后端联调、功能测试、算法参数调优 | 5 | 前后端基本完成 |
| 文档与答辩 | 技术文档、用户手册、答辩PPT | 3 | 系统基本稳定 |
| **合计** | — | **33人天** | — |

#### 4.1.2 进度可行性分析

**这是本报告最重要的分析之一。** 33人天是否能在6周（30个工作日）内完成？

**乐观场景（3人团队，全程并行）：**
- 第1周：环境搭建 + MVP验证（3人天 × 1人 = 3天）
- 第2—3周：后端核心开发（12人天 ÷ 2人并行 = 6天）
- 第3—4周：前端开发（可用Mock先行，10人天 ÷ 1.5人 = 7天）
- 第5周：联调测试（5人天 ÷ 2.5人 = 2天）
- 第6周：文档与答辩（3人天 ÷ 2人 = 1.5天）
- 总耗时约 4.5周，**有约1.5周缓冲**。

**悲观场景（2人团队，串行开发）：**
- 后端必须先完成 → 前端才能联调
- 关键路径：MVP(3天) + 后端(12天) + 联调(5天) = 20天
- 前端可与后端后期并行：10天
- 文档可穿插进行：3天
- 总耗时：max(20, 10) + 3 ≈ 23天 + 不可预见阻塞 ≈ 27—30天
- **缓冲约0—3天，非常紧张。**

**关键路径：** 环境搭建(3天) → 后端Agent引擎(5天) → 推荐算法流水线(7天) → 前后端联调(5天) → 文档(3天) = **23个工作日**。

**进度结论：** 3人团队可行且有缓冲；2人团队偏紧但可完成。**关键前提是：第1周MVP验证必须通过，否则后续全部推迟。**

#### 4.1.3 与替代方案的成本对比

| 方案 | 预估开发成本 | 技术深度 | 答辩展示效果 |
|------|-------------|----------|-------------|
| **本方案（Agent + PostGIS + 推荐流水线）** | 33人天 | 高 | 优秀 |
| 简化方案A：LLM直接推荐 + 高德API展示 | 15人天 | 低 | 一般 |
| 简化方案B：LLM + 高德API + 简单距离排序（无聚类/TSP） | 20人天 | 中 | 良好 |

本方案成本约为简化方案A的2倍，但技术深度和展示效果显著优于简化方案。考虑到课设评分对技术含量的权重，**投入额外成本是合理的**。

### 4.2 运行成本分析

| 成本项 | 单价 | 预估用量 | 月费用 |
|--------|------|----------|--------|
| DashScope API（qwen-plus） | 0.008元/千tokens | 约50万tokens（演示期） | ≈ 4元 |
| 高德地图API | 免费（个人开发者日配额30万次） | 远低于配额 | 0元 |

## 5 操作可行性分析

### 5.1 用户使用场景分析

**场景一：初次旅行规划**

用户输入"下周末去杭州两天，喜欢历史文化，不想走太多路" → 系统解析意图 → 检索POI → 五步推荐流水线 → 生成两日行程 → 地图标注景点和路线。

**可行性判断：** 该场景是系统的核心路径。意图解析（LLM Function Calling，成熟）、空间分析（PostGIS，成熟）、路线可视化（Leaflet，成熟）各环节技术风险低。**可行。**

**场景二：多轮对话调整**

用户说"第二天下午换个湖边景点" → 系统恢复对话状态 → 在已有行程基础上筛选新POI → 更新地图。

**可行性判断：** 多轮对话依赖LangGraph的Checkpoint机制（支持PostgreSQL后端）。**可行，但需在MVP阶段验证Checkpoint与PostGIS的兼容性。**

**场景三：异常与边界**

用户输入无效目的地、不合理天数、或超出系统能力范围的请求（如"规划环球影城到迪士尼的路线"但两者不在同一城市）→ Router Agent识别异常 → 返回友好提示。

**可行性判断：** LLM的意图分类能力足以处理此类边界情况。**可行。**

### 5.2 系统可用性评估

| 维度 | 评估 | 说明 |
|------|------|------|
| 学习成本 | 低 | 自然语言交互，用户零学习成本 |
| 响应体验 | 中上 | SSE流式推送，首token延迟2—3秒 |
| 错误容忍 | 中 | LLM降级机制保障基本可用，但降级后推荐质量大幅下降 |
| 结果可靠性 | 中 | LLM输出存在不确定性，需通过结构化约束降低幻觉风险 |

### 5.3 部署和维护可行性

**部署：** Docker Compose一键编排（Nginx + FastAPI + PostgreSQL），实现环境一致性。部署命令`docker compose up -d`，适合课设演示。

**维护：** 分层架构下各模块独立可替换。日常维护主要是API密钥管理和日志监控。PostGIS空间索引需定期维护，但课设周期内数据量小，维护成本可忽略。

## 7 可行性结论与建议

### 7.1 综合评价

| 维度 | 结论 | 核心依据 |
|------|------|----------|
| 技术可行性 | ✅ 可行 | 核心技术栈成熟；最大风险在五步流水线的集成调试和DBSCAN参数调优 |
| 经济可行性 | ✅ 可行 | 开发33人天，运行成本趋近于零，教学效益显著 |
| 进度可行性 | ⚠️ 可行但偏紧 | 3人团队有缓冲（约1.5周）；2人团队缓冲极少（0—3天） |
| 操作可行性 | ✅ 可行 | Docker一键部署，自然语言交互零门槛 |
| 社会因素 | ✅ 可行 | 开源合规，法律风险低 |

### 7.2 结论

**项目总体可行，但存在明确的前提条件：**

1. **进度可行性的前提：** 第1周必须完成MVP验证（LangGraph + DashScope + PostGIS最小闭环），否则后续全部推迟。如果团队在第1周末仍无法跑通基础链路，建议降级为简化方案B（去掉DBSCAN聚类和TSP优化，改用简单距离排序）。

2. **技术可行性的前提：** 五步推荐流水线需要在第2—3周集中攻克。如果DBSCAN聚类效果持续不理想，可降级为基于地理区域的简单分天（如按行政区划分组），保证核心功能可交付。

3. **LangGraph的不确定性：** 作为成熟度最低的技术组件，需锁定版本、预留排障时间。如果LangGraph在开发过程中出现严重Bug且无法快速解决，应果断降级为LangChain简单Chain + 手动状态管理。

### 7.3 建议

| 建议 | 优先级 | 理由 |
|------|--------|------|
| **第1周完成MVP验证** | 最高 | 尽早暴露集成问题，决定后续方案走向 |
| **后端优先于前端** | 高 | 五步流水线是核心难点，应先于前端地图开发完成并验证 |
| **锁定LangGraph版本** | 高 | 避免开发期间API变动带来的额外排障成本 |
| **准备降级预案** | 中 | DBSCAN效果不佳 → 按区域分天；LangGraph出问题 → 降级为简单Chain |
| **DBSCAN参数可配置化** | 中 | 避免硬编码eps值，支持按城市动态调整 |
| **准备固定演示场景** | 中 | 确保演示路径上每个环节效果稳定可控 |
