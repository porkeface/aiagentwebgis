# 可行性研究报告

## 基于AI Agent的旅游路线智能规划与景点空间推荐系统

---

## 1 引言

### 1.1 编写目的

本报告旨在回答核心问题：**在课程设计周期约束下，构建融合AI Agent与WebGIS的旅游路线规划系统是否可行？关键瓶颈在哪里？**

报告从技术、经济、进度、操作、社会因素五个维度进行量化评估，并给出明确的前提条件和降级预案。

### 1.2 项目背景

旅游路线规划是多目标决策问题——用户需要综合景点选择、空间距离、时间分配、个人偏好等因素。现有方案存在两类不足：

- **传统旅游平台**：提供景点列表和攻略，规划依赖用户手动筛选拼接，不具备空间智能
- **通用LLM对话工具**：能生成旅行建议文本，但缺乏空间分析能力——无法计算真实距离、无法空间聚类分天、无法路径优化

本项目的核心思路是：**将LLM的意图理解能力与GIS的空间分析能力通过Agent机制结合**，构建一个能看地图、会做空间分析的AI旅行规划系统。

### 1.3 定义与缩略语

| 术语 | 定义 |
|------|------|
| AI Agent | 基于LLM的智能代理，具备意图理解、工具调用、多步推理能力 |
| LangGraph | 基于图结构的LLM Agent编排框架 |
| PostGIS | PostgreSQL空间数据库扩展 |
| POI | 兴趣点（Point of Interest） |
| TSP | 旅行商问题（Traveling Salesman Problem） |
| DBSCAN | 基于密度的空间聚类算法 |
| SSE | 服务器发送事件（Server-Sent Events） |
| ReAct | Reasoning + Acting，Agent交替思考与调用工具的模式 |

## 2 可行性研究的前提

### 2.1 问题定义

本系统的核心问题：**如何让用户用自然语言描述旅行需求，系统自动完成景点筛选、空间分天和路线优化，并在地图上可视化？**

| 子问题 | 核心挑战 | 难度 | 说明 |
|--------|----------|------|------|
| 意图解析 | 从非结构化文本中提取目的地、天数、偏好等参数 | 中 | LLM意图分类已较成熟 |
| 空间分析与推荐 | 基于地理位置进行POI分天和路径优化 | 高 | 涉及多环节串联和参数调优，是核心难点 |
| 结果可视化与交互 | 高德地图标注、路线绘制、对话联动 | 中 | 高德JS API生态成熟 |

### 2.2 系统目标

| 编号 | 目标 | 验收标准 |
|------|------|----------|
| G1 | 自然语言意图解析 | 输入"去杭州玩两天，喜欢历史文化"，正确提取目的地、天数、偏好 |
| G2 | 空间POI检索 | 基于目的地和偏好，调用高德API + 本地PostGIS返回相关景点 |
| G3 | 多日行程规划 | 流水线输出合理多日行程（搜索→评分→分天→路径→验证） |
| G4 | 路线可视化 | 高德地图标注景点、绘制连线，与对话联动 |
| G5 | 多轮对话 | 支持在已有行程基础上修改调整 |
| G6 | 流式响应 | SSE推送Agent推理过程，首token延迟<3秒 |

### 2.3 系统范围

**包含：** 前端对话+地图可视化 + 后端Agent引擎 + 空间规划流水线 + PostGIS数据库 + 管理后台。

**不包含：** 支付与订单系统、UGC社交功能、移动端原生适配、OAuth第三方登录。

### 2.4 约束条件

| 约束 | 具体内容 | 影响 |
|------|----------|------|
| 时间 | 课程设计周期，有效编码约4—5周 | 关键 |
| 人力 | 学生团队 | 关键 |
| 技术 | LLM依赖外部API（网络延迟、配额限制） | 中等 |
| 数据 | POI数据依赖高德API + PostGIS本地库 | 低 |

## 3 技术可行性分析

### 3.1 技术架构

基于对实际技术栈的分析，系统采用四层架构：

```
┌──────────────────────────────────────────────────┐
│               前端可视化层                        │
│  Vue 3 + Vite + TypeScript + 高德JS API         │
│  Element Plus + Pinia                           │
│  职责：对话交互、地图可视化、行程管理             │
├──────────────────────────────────────────────────┤
│               API服务层                          │
│  FastAPI + SQLAlchemy 2.0 + GeoAlchemy2         │
│  职责：RESTful API、SSE流式推送、JWT认证         │
├──────────────────────────────────────────────────┤
│               AI Agent层                         │
│  LangGraph StateGraph + ChatOpenAI              │
│  意图分类→规划流水线/Agent循环→摘要输出          │
│  7个工具：搜索/地理编码/评分/路线/分天/提交       │
├──────────────────────────────────────────────────┤
│               数据存储层                         │
│  PostgreSQL + PostGIS  │  高德API  │  LLM API   │
└──────────────────────────────────────────────────┘
```

### 3.2 关键技术环节可行性

#### 3.2.1 LLM工具调用 —— 可行

系统采用ChatOpenAI兼容接口，支持DashScope（通义千问）和DeepSeek。LLM的Function Calling已较成熟，Agent可调用7个工具完成复杂规划任务。参数校验层和自动重试机制确保工具调用的鲁棒性。

#### 3.2.2 规划流水线 —— 核心创新点

这是项目的核心——不使用LLM逐步推理调用工具，而是在意图分类后进入一个**确定性流水线**：

1. **并行搜索**：5类关键词（风景名胜、博物馆、公园、寺庙道观、特色街区）并行调用高德POI搜索
2. **语义去重**：移除同名/近名重复POI
3. **多因子评分**：基于偏好匹配度、距离、评分、热度综合打分
4. **时长估算**：动态估算每个POI的游览时长，计算每日容量
5. **地理分区**：DBSCAN聚类分天 + 链式排序确保跨日连续
6. **障碍检测**：高德驾车距离 vs 直线距离对比，发现湖/山阻隔
7. **每日路线**：并行调用高德路径API规划每日最优游览顺序
8. **跨日交换**：haversine距离检测边界POI的跨日重分配
9. **验证提交**：Critic校验+数据持久化
10. **LLM摘要**：生成自然语言行程介绍

每步单独可行，串联后的集成调试是核心挑战。流水线设计避免了LLM在规划过程中产生幻觉，规划质量的确定性远高于纯LLM驱动的方案。

#### 3.2.3 LLM适配 —— 可行

通过ChatOpenAI统一接口同时支持DashScope和DeepSeek，管理员在后台界面切换供应商后配置热重载生效，无需重启服务。

#### 3.2.4 前端地图 —— 成熟可靠

采用高德JS API 2.0，国内地图数据准确，中文标注完善。POI数量通常在20—50个，Marker渲染性能完全满足。

### 3.3 技术风险评估

| 风险 | 概率 | 影响 | 缓解策略 |
|------|------|------|----------|
| LLM响应超时 | 低 | 中 | SSE流式推送 + 60s超时 |
| 高德API日配额耗尽 | 低 | 中 | 本地PostGIS缓存 |
| DBSCAN参数不适配城市 | 中 | 中 | 动态eps + 障碍检测兜底 |
| LLM工具调用参数错误 | 中 | 低 | 参数校验+自动重试 |

## 5 操作可行性分析

### 5.1 用户场景

**场景一：初次旅行规划。** 用户输入"下周末去杭州两天，喜欢历史文化" → 系统分类意图 → 执行规划流水线 → 生成两日行程 → 高德地图标注。

**场景二：地图选点规划。** 用户在首页地图上先手动选择感兴趣POI → 发送"帮我规划这些点的行程" → 系统走预选POI快速通道 → 跳过搜索直接分天路线。

**场景三：管理后台配置。** 管理员在后台切换LLM供应商、修改API Key、管理高德Key → 配置热重载立即生效。

### 5.2 部署可行性

- **开发模式**：前端`npm run dev`（Vite）+ 后端`uvicorn --reload`
- **运行环境**：Windows 10/11，需要本地PostgreSQL 15+ + PostGIS
- **启动步骤**：安装依赖 → 创建数据库 → 配置.env → 启动前后端
