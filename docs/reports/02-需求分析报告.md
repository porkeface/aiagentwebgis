## 1 引言

### 1.1 编写目的

本报告旨在对"基于AI Agent的旅游路线智能规划与景点空间推荐系统"进行全面、系统的需求分析，明确系统的功能需求、非功能需求、接口需求及运行环境约束，为后续的系统设计、编码实现及测试验收提供基础依据。

本报告的预期读者包括：项目组成员、课程指导教师、评审专家，以及对系统设计方案感兴趣的相关技术人员。

### 1.2 项目背景

随着我国旅游业的持续蓬勃发展，自由行已成为主流出行方式。据文化和旅游部统计数据显示，国内旅游人次逐年攀升，用户对个性化、智能化旅行规划的需求日益迫切。然而，传统旅游规划方式存在以下痛点：

1. **信息过载**：用户面对海量景点信息，难以高效筛选符合自身偏好的目的地；
2. **空间分析缺失**：现有推荐系统多基于属性匹配，缺乏对地理空间关系的深度分析，导致推荐景点分布零散、路线规划不合理；
3. **交互门槛高**：传统规划工具要求用户手动操作地图、设定筛选条件，学习成本高，交互体验差；
4. **缺乏个性化**：难以综合考虑用户偏好、体力约束、预算限制等多维度因素进行智能决策。

近年来，大语言模型（LLM）与AI Agent技术的突破性进展，为解决上述问题提供了全新的技术路径。与此同时，WebGIS（Web地理信息系统）的成熟使空间分析能力得以在浏览器端高效呈现。本项目创新性地将AI Agent与WebGIS深度融合，构建"能看地图、会做空间分析的AI旅行规划师"，用户仅需以自然语言描述旅行需求，系统即可自动完成空间分析、智能推荐与路线规划，并在地图上直观可视化展示结果。

## 2 任务概述

### 2.1 系统目标

本系统的总体目标为：构建一个融合AI Agent智能推理与WebGIS空间分析能力的旅游路线智能规划系统，实现用户自然语言驱动的端到端旅行方案生成。

具体目标分解如下：

**G1 — 自然语言理解目标**：系统能够准确解析用户的自然语言输入，提取目的地、出行天数、偏好类型、同行人员构成、预算范围等多维度旅行意图信息。

**G2 — 空间智能分析目标**：系统能够基于PostGIS空间数据库执行POI检索、缓冲区分析、空间聚类等操作，为推荐与规划提供空间决策支持。

**G3 — 个性化推荐目标**：系统能够综合用户偏好、空间距离、景点属性等因素，通过多因子评分与多样性重排策略，生成兼顾相关性与多样性的景点推荐列表。

**G4 — 路线自动规划目标**：系统能够基于时间约束、空间距离、景点游览时长等因素，自动生成合理的多日游览路线，并在地图上进行可视化展示。

**G5 — 多轮交互优化目标**：系统支持多轮对话式交互，Agent能够保持上下文记忆，根据用户反馈持续迭代优化旅行方案。

**G6 — 多Agent协作目标**：系统采用Router→Planner→Formatter的多Agent协作架构，通过共享State白板模式实现Agent间的消息流转与协同推理。

### 2.2 用户特点

本系统的目标用户群体及其特征如下：

| 用户类别 | 特征描述 | 核心需求 |
|---------|---------|---------|
| **自由行旅客** | 18-45岁，具备基本网络操作能力，追求个性化旅行体验 | 个性化路线规划、避开热门景点拥挤 |
| **家庭出游者** | 携带老人或儿童的家庭用户，关注出行舒适度 | 考虑老人、小孩体力约束的舒适路线，合理安排休息点 |
| **背包客** | 预算敏感型旅行者，注重性价比与效率 | 预算优化、距离最优路线、经济型推荐 |
| **旅游爱好者** | 频繁出行，对目的地有一定了解，希望深度探索 | 深度景点推荐、小众景点发现、主题化路线 |

用户共性特征：具备基本的Web应用操作能力，能够通过自然语言描述旅行需求，期望系统提供直观、易用的地图可视化交互体验。

### 2.3 假定和约束

**假定条件：**

- A1：用户在对话中能够提供基本可解析的旅行需求描述（至少包含目的地信息）；
- A2：系统运行所需的第三方服务（DashScope LLM API、OSRM路径规划服务等）在系统使用期间保持可用；
- A3：PostGIS空间数据库中已预置目标城市的基础POI数据；
- A4：用户网络环境稳定，能够支持地图瓦片的正常加载。

**约束条件：**

- C1（时间约束）：项目开发周期约6周，需合理划分P0/P1/P2优先级，确保核心功能在答辩前完成；
- C2（资源约束）：LLM API调用存在Token费用，需控制单次请求的Token消耗量；
- C3（技术约束）：通义千问DashScope的API调用频率有限制（QPS限制），需实现请求排队与限流机制；
- C4（数据约束）：POI数据的完整性和准确性直接影响推荐质量，数据来源受限于开放数据；
- C5（硬件约束）：开发与测试阶段受限于开发机器的计算资源，大规模空间分析需考虑性能边界。

## 3 需求规定

### 3.1 功能需求

#### 3.1.1 自然语言意图解析模块

**功能标识**：FR-001

**功能描述**：接收用户的自然语言输入，调用LLM进行意图识别与实体抽取，输出结构化的旅行意图对象。

**输入**：

| 输入项 | 类型 | 必选 | 说明 |
|--------|------|------|------|
| user_input | String | 是 | 用户自然语言文本，如"我想带父母去杭州玩三天，预算5000左右" |
| context | JSON | 否 | 多轮对话上下文，用于指代消解与意图延续 |

**处理**：

1. 将用户输入与对话上下文组装为Prompt模板；
2. 调用DashScope LLM API进行意图解析；
3. 提取结构化字段：目的地（destination）、天数（days）、偏好标签（preferences）、同行人员（companions）、预算范围（budget）、出行时间（travel_time）等；
4. 对缺失字段生成追问策略，引导用户补充关键信息；
5. 输出标准化的意图JSON对象。

**输出**：

```json
{
  "destination": "杭州",
  "days": 3,
  "preferences": ["文化古迹", "自然风光"],
  "companions": {"type": "family", "members": ["adult", "elderly"]},
  "budget": {"min": 3000, "max": 5000, "currency": "CNY"},
  "travel_time": null,
  "missing_fields": ["travel_time"]
}
```

**优先级**：P0

#### 3.1.2 空间POI检索模块

**功能标识**：FR-002

**功能描述**：根据意图解析结果中的目的地信息，在PostGIS空间数据库中检索目标区域范围内的景点POI数据，并将结果标注在WebGIS地图上。

**输入**：

| 输入项 | 类型 | 必选 | 说明 |
|--------|------|------|------|
| destination | String | 是 | 目的地名称或地理编码 |
| search_radius | Float | 否 | 搜索半径（米），默认50000m |
| category_filter | String[] | 否 | 景点类别过滤条件 |

**处理**：

1. 对目的地名称进行地理编码（Geocoding），获取中心坐标（lng, lat）；
2. 构建PostGIS空间查询SQL，以中心坐标为圆心、search_radius为半径进行空间范围过滤；
3. 应用category_filter进行属性过滤；
4. 查询结果包含：景点名称、坐标、类别、评分、开放时间、门票价格等；
5. 将POI数据序列化为GeoJSON格式返回前端。

**输出**：

```json
{
  "center": [120.15, 30.28],
  "total": 42,
  "pois": [
    {
      "id": 1,
      "name": "西湖",
      "category": "自然风光",
      "location": [120.15, 30.25],
      "rating": 4.8,
      "visit_duration": 180,
      "ticket_price": 0,
      "opening_hours": "全天开放"
    }
  ]
}
```

**优先级**：P0

#### 3.1.3 空间感知推荐模块

**功能标识**：FR-003

**功能描述**：基于用户偏好和空间位置信息，通过五步推荐流水线（空间过滤→多因子评分→MMR多样性重排→DBSCAN聚类分天→TSP路线优化），生成个性化的景点推荐方案。

**输入**：

| 输入项 | 类型 | 必选 | 说明 |
|--------|------|------|------|
| pois | GeoJSON[] | 是 | 候选景点集合 |
| intent | JSON | 是 | 用户意图对象（含偏好、天数、预算等） |
| weights | JSON | 否 | Agent动态生成的评分权重向量 |

**处理**：

1. **空间过滤**：基于用户住宿位置或指定起点，利用PostGIS计算各景点的空间可达性，剔除不可达景点；
2. **多因子评分**：综合计算每个景点的评分，评分因子包括——偏好匹配度（由Agent动态赋权）、空间距离得分、热度得分、性价比得分，公式为：$S(p) = \sum_{i} w_i \cdot f_i(p)$，其中 $w_i$ 为Agent根据用户画像动态调整的权重，$f_i(p)$ 为各因子的归一化分值；
3. **MMR多样性重排**：采用Maximal Marginal Relevance策略，在保证相关性的同时最大化推荐列表的类别多样性，避免推荐景点类型同质化；
4. **DBSCAN聚类分天**：对推荐景点按空间密度进行DBSCAN聚类，将地理邻近的景点归为同一天的游览组，确保每天的游览景点空间集中，减少无效交通时间；
5. **TSP路线优化**：对每天的景点组，采用贪最近邻+2-opt改进的TSP近似算法，规划当天最优游览顺序。

**输出**：

```json
{
  "days": [
    {
      "day": 1,
      "cluster_center": [120.15, 30.25],
      "pois": [
        {"id": 1, "name": "西湖", "score": 0.92, "reason": "高偏好匹配+核心景区"},
        {"id": 3, "name": "雷峰塔", "score": 0.85, "reason": "紧邻西湖，文化古迹偏好匹配"}
      ],
      "route_order": [1, 3, 7],
      "total_distance_km": 8.5,
      "total_duration_min": 420
    }
  ],
  "rejected_pois": [{"id": 15, "reason": "空间不可达"}]
}
```

**优先级**：P0（核心推荐）、P1（DBSCAN聚类+TSP优化为P1）

#### 3.1.4 智能路线规划模块

**功能标识**：FR-004

**功能描述**：综合考虑时间约束、空间距离、景点游览时长等因素，生成完整的多日游览路线方案，并在地图上以连线方式可视化展示。

**输入**：

| 输入项 | 类型 | 必选 | 说明 |
|--------|------|------|------|
| route_pois | JSON | 是 | 按天分组的景点序列（含游览顺序） |
| constraints | JSON | 否 | 额外约束条件（如每日最大步行距离） |
| start_location | Coordinate | 否 | 出发点坐标 |

**处理**：

1. 对每天的景点序列，调用OSRM路径规划API获取相邻景点间的实际道路距离和通行时间；
2. 校验路线是否满足时间约束（每日游览总时长 ≤ 用户可接受时长）；
3. 若约束不满足，触发路线调整策略（移除末位景点或建议增加天数）；
4. 生成包含逐段路线信息的完整路线方案；
5. 将路线数据序列化为GeoJSON LineString格式供前端渲染。

**输出**：

```json
{
  "day": 1,
  "route": {
    "type": "LineString",
    "coordinates": [[120.15, 30.25], [120.16, 30.26], ...]
  },
  "segments": [
    {
      "from": "西湖",
      "to": "雷峰塔",
      "distance_km": 1.2,
      "travel_time_min": 15,
      "visit_duration_min": 90
    }
  ],
  "summary": {
    "total_distance_km": 8.5,
    "total_travel_time_min": 60,
    "total_visit_time_min": 360
  },
  "feasible": true
}
```

**优先级**：P0

#### 3.1.5 Agent协作编排模块

**功能标识**：FR-005

**功能描述**：基于LangGraph StateGraph构建多Agent协作架构，实现Router（意图路由）→ Planner Agent（核心推理大脑）→ Formatter（格式化输出）的协同工作流。Agent间通过共享State白板模式进行消息流转，而非简单流水线传递。

**输入**：

| 输入项 | 类型 | 必选 | 说明 |
|--------|------|------|------|
| user_message | String | 是 | 用户当前轮次的自然语言输入 |
| state | StateGraph | 是 | 共享状态对象（含历史对话、当前意图、中间推理结果等） |

**处理**：

1. **Router Agent**：接收用户输入，执行意图分类（新规划/方案调整/信息查询/闲聊），将请求路由至对应处理节点；
2. **Planner Agent**：核心推理节点，采用ReAct模式，循环执行"思考→选择工具→调用工具→观察结果→继续思考"的推理链，可选工具包括：空间查询工具、POI检索工具、路线规划工具、天气查询工具等；
3. **Formatter Agent**：将Planner的推理结果格式化为面向用户的结构化输出（含文本描述+地图数据）；
4. **共享State**：所有Agent读写同一个State对象，State包含：`messages`（消息历史）、`intent`（当前意图）、`candidates`（候选景点）、`plan`（当前方案）、`tool_results`（工具调用结果）等字段；
5. **Checkpoint持久化**：利用LangGraph Checkpoint机制，将State快照持久化至PostgreSQL，支持多轮对话状态恢复。

**输出**：

```json
{
  "agent_flow": ["Router", "Planner", "Planner(Tool:spatial_query)", "Planner(Tool:recommend)", "Formatter"],
  "final_state": {
    "messages": [...],
    "intent": {...},
    "recommendations": [...],
    "route_plan": {...}
  },
  "response": {
    "text": "根据您的偏好，我为您推荐以下三天杭州深度游方案...",
    "map_data": {...}
  }
}
```

**优先级**：P0（基础协作）、P1（Agent消息流转可视化）

#### 3.1.6 地图可视化模块

**功能标识**：FR-006

**功能描述**：基于Leaflet.js构建WebGIS地图界面，实现景点标注、路线连线、空间分析结果的多层可视化展示，并支持用户交互式地图操作。

**输入**：

| 输入项 | 类型 | 必选 | 说明 |
|--------|------|------|------|
| map_data | GeoJSON | 是 | 待展示的地理数据（景点Point、路线LineString、缓冲区Polygon等） |
| layer_type | String | 是 | 图层类型：poi/route/analysis |

**处理**：

1. 初始化Leaflet地图实例，加载底图瓦片（可选OpenStreetMap/高德地图）；
2. 根据layer_type创建对应的GeoJSON图层：
   - POI图层：以自定义Marker标注景点位置，点击弹出InfoWindow显示景点详情；
   - 路线图层：以不同颜色的Polyline连接每日路线，线段粗细反映距离权重；
   - 分析图层：展示缓冲区（Circle）、聚类区域（GeoJSON Polygon）等空间分析结果；
3. 支持地图交互操作：缩放、平移、图层切换、景点点击、路线高亮；
4. P1功能：支持拖拽景点Marker调整游览顺序，实时重算路线。

**输出**：

- 交互式地图界面，支持多图层叠加显示
- 景点信息弹窗（含名称、评分、游览时长、推荐理由等）
- 路线信息面板（含总距离、总时长、分段详情）

**优先级**：P0（基础可视化）、P1（拖拽编辑+空间分析图层）

#### 3.1.7 多轮对话管理模块

**功能标识**：FR-007

**功能描述**：管理用户与系统之间的多轮对话交互，支持上下文记忆、指代消解、方案迭代优化等功能。

**输入**：

| 输入项 | 类型 | 必选 | 说明 |
|--------|------|------|------|
| message | String | 是 | 用户当前轮次输入 |
| session_id | UUID | 是 | 会话唯一标识 |

**处理**：

1. 根据session_id从Checkpoint存储中恢复对话State；
2. 将当前用户消息追加至State的messages列表；
3. 执行指代消解（如"把第二个景点换成灵隐寺"中的"第二个景点"）；
4. 将完整消息历史传入Agent协作流程；
5. 更新State并通过Checkpoint持久化；
6. 返回Agent的响应消息。

**输出**：

```json
{
  "session_id": "uuid-xxx",
  "turn": 3,
  "response": {
    "text": "好的，已将第二天的景点'雷峰塔'替换为'灵隐寺'，重新规划了路线...",
    "updated_plan": {...},
    "map_data": {...}
  }
}
```

**优先级**：P0

#### 3.2.2 安全性需求

- S1：API接口实施JWT Token认证，防止未授权访问；
- S2：LLM API密钥通过环境变量管理，不得硬编码于源代码中；
- S3：用户输入进行注入攻击防护（SQL注入、Prompt注入）；
- S4：前后端通信采用HTTPS加密传输；
- S5：实施API请求频率限制（Rate Limiting），防止恶意调用。

#### 3.2.3 可靠性需求

- R1：系统可用性 ≥ 95%（课程演示期间）；
- R2：LLM API调用失败时自动重试（最多3次），重试失败后返回友好降级响应；
- R3：数据库操作支持事务一致性，避免数据不一致状态；
- R4：前端在地图加载失败、网络中断等异常场景下显示明确的错误提示。

#### 3.2.4 可维护性需求

- M1：代码遵循PEP 8（后端）和ESLint（前端）编码规范；
- M2：模块间采用松耦合设计，核心模块提供清晰的接口定义；
- M3：关键业务逻辑配备单元测试，目标覆盖率 ≥ 80%；
- M4：提供API文档（Swagger/OpenAPI 3.0自动生成）。

#### 3.2.5 可扩展性需求

- E1：LLM层采用适配器模式，支持切换至其他LLM提供商（OpenAI、本地模型等）；
- E2：工具注册采用插件化机制，便于扩展新的Agent工具；
- E3：空间数据库支持多城市数据扩展，不影响现有查询逻辑；
- E4：前端地图支持底图源切换（OSM/高德/天地图）。
