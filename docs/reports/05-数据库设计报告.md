## 1 引言

### 1.1 编写目的

本报告对系统数据库进行全面设计，涵盖需求分析、概念结构（E-R图）、逻辑结构、物理结构、索引策略及安全设计。

### 1.2 项目背景

系统需要管理大量POI空间数据，支持复杂空间查询与分析操作，同时管理用户信息、行程规划、对话会话等业务数据。选择PostgreSQL+PostGIS作为数据存储引擎。

### 1.3 定义与缩略语

| 术语 | 说明 |
|------|------|
| POI | Point of Interest，兴趣点 |
| PostGIS | PostgreSQL空间数据库扩展 |
| GIST | Generalized Search Tree，通用搜索树索引 |
| JSONB | PostgreSQL二进制JSON类型 |
| SRID | Spatial Reference Identifier，空间参考标识符（4326=WGS84） |
| FK | Foreign Key，外键 |
| PK | Primary Key，主键 |
| 3NF | Third Normal Form，第三范式 |

## 2 需求分析

### 2.1 数据实体

**(1) POI（兴趣点）**：景点、餐厅、酒店等地理实体。属性：名称、分类、坐标（GEOMETRY POINT）、评分、标签、描述、扩展数据（JSONB）。

**(2) User（用户）**：系统注册使用者。属性：用户名、密码哈希、昵称、邮箱、管理员标识。

**(3) Trip（行程）**：Agent生成的旅行计划。属性：标题、目标城市、起止日期、状态。

**(4) TripDay（日程）**：行程下按天拆分。属性：天数序号、日期、路线折线、总计距离/时间。

**(5) TripDayPOI（日程景点）**：每日具体景点安排。属性：排序、到达/离开时间、评分。

**(6) ChatSession（对话会话）**：用户与Agent对话记录。属性：thread_id、消息JSON、Agent状态JSON。

### 2.2 数据量估算

| 数据表 | 预估量 | 增长趋势 |
|--------|--------|---------|
| users | <500条 | 缓慢 |
| pois | 5,000—15,000条 | 中速 |
| trips | <1,000条 | 中速 |
| trip_days | <5,000条 | 与trip同步 |
| trip_day_pois | <20,000条 | 较快 |
| chat_sessions | <3,000条 | 中速 |

总数据量万级，属于中小型应用，PostgreSQL完全胜任。

### 2.3 核心查询需求

| 编号 | 查询类型 | 说明 | 技术依赖 |
|------|---------|------|---------|
| Q1 | 空间范围查询 | 某城市矩形区域内POI | ST_Within + GIST索引 |
| Q2 | 空间距离查询 | 距某坐标N公里内POI | ST_DWithin + GIST索引 |
| Q3 | 类别+评分查询 | 按城市+类别筛选，评分排序 | 复合B-tree索引 |
| Q4 | 标签查询 | 按标签数组筛选 | GIN索引 on TEXT[] |
| Q5 | 行程关联查询 | 用户→行程→日→POI的层级JOIN | 外键索引 |
| Q6 | 会话查询 | 按UUID查询对话历史 | UUID索引 |

## 3 概念结构设计

### 3.1 E-R图

```
┌──────────┐        ┌──────────────┐        ┌──────────┐
│   User   │1──────N│     Trip     │1──────N│ TripDay  │
└──────────┘        └──────────────┘        └──────────┘
     │                     │                1     │
     │1                    │                │     │N
     │                     │                │     │
     │N                    │N               │  ┌──────────────┐
┌──────────────┐     ┌──────────┐           │  │ TripDayPOI   │
│ ChatSession  │     │   POI    │◄──────────┘  └──────────────┘
└──────────────┘     └──────────┘             N     │
                                                   │1
                                              ┌──────────┐
                                              │   POI    │
                                              └──────────┘
```

### 3.2 实体关系说明

| 关系 | 类型 | 说明 |
|------|------|------|
| User → ChatSession | 1:N | 一个用户有多个对话会话 |
| User → Trip | 1:N | 一个用户有多个行程 |
| Trip → TripDay | 1:N | 一个行程有多个日程（天） |
| TripDay → TripDayPOI | 1:N | 一天有多个POI安排 |
| TripDayPOI → POI | N:1 | 多个日程条目可引用同一POI |

## 4 逻辑结构设计

### 4.1 关系模式（3NF）

所有表满足第三范式，每个非主属性完全函数依赖于主键且不传递依赖。

**users**(id, username, hashed_password, nickname, email, is_admin, created_at)

**pois**(id, name, category, city, location:GEOMETRY(POINT,4326), rating, tags:TEXT[], description, opening_hours, avg_visit_duration, review_count, extra_data:JSONB, created_at)

**trips**(id, user_id:FK→users, title, city, start_date, end_date, status, notes, created_at, updated_at)

**trip_days**(id, trip_id:FK→trips, day_number, date, notes, polyline, segments_json, total_distance_km, total_duration_min)

**trip_day_pois**(id, trip_day_id:FK→trip_days, poi_id:FK→pois, sort_order, arrival_time, departure_time, score, notes)

**chat_sessions**(id, user_id:FK→users, thread_id, title, messages_json:JSONB, agent_state_json:JSONB, created_at, updated_at)

### 4.2 级联策略

所有外键使用`ON DELETE CASCADE`——删除用户时自动删除其所有旅程/会话、删除行程时自动删除其所有日程和POI关联。

## 7 索引设计

### 7.1 索引总览

| 索引名称 | 表 | 类型 | 列 | 用途 |
|---------|-----|------|-----|------|
| ix_pois_location_gist | pois | GIST | location | 空间范围/距离查询加速（核心） |
| ix_pois_city_category_rating | pois | B-tree复合 | (city, category, rating) | 按城市+类别查询并按评分排序 |
| uq_pois_extra_amap_id | pois | B-tree部分唯一 | (extra_data->amap_id) | 防止同一高德POI重复入库 |
| ix_users_username | users | B-tree唯一 | username | 用户登录查找 |
| ix_users_email | users | B-tree | email | 邮箱查找 |
| ix_trips_user_id | trips | B-tree | user_id | 用户行程列表查询 |
| ix_trips_city | trips | B-tree | city | 按城市筛选行程 |
| ix_trips_status | trips | B-tree | status | 按状态筛选 |
| ix_trip_days_trip_id | trip_days | B-tree | trip_id | 行程→日程JOIN |
| ix_trip_day_pois_trip_day_id | trip_day_pois | B-tree | trip_day_id | 日程→POI关联JOIN |
| ix_trip_day_pois_poi_id | trip_day_pois | B-tree | poi_id | POI反查日程 |
| ix_chat_sessions_user_id | chat_sessions | B-tree | user_id | 用户会话列表 |
| ix_chat_sessions_thread_id | chat_sessions | B-tree | thread_id | UUID查找会话 |
| uq_chat_sessions_user_thread | chat_sessions | Unique | (user_id, thread_id) | 防止重复会话 |

### 7.2 关键索引设计说明

**GIST空间索引**：`ix_pois_location_gist`是系统的性能核心。所有空间查询（ST_DWithin、ST_Within、ST_MakeEnvelope）都依赖此索引。不使用GIST索引的情况下，每次空间查询需要全表扫描O(n)，使用后为O(log n)。

**复合索引**：`ix_pois_city_category_rating`覆盖最常见查询模式——"某城市的某类POI按评分排序"。三列复合的顺序为(city, category, rating)，支持前缀匹配（单独按city或city+category查询也能用到此索引）。

**部分唯一索引**：`uq_pois_extra_amap_id`仅在extra_data->amap_id不为空时生效。避免submit_plan时并发写入导致同一高德POI出现多条记录。

## 8 安全设计

### 8.1 访问控制

| 安全措施 | 实现方式 |
|---------|---------|
| 数据库认证 | 用户名+密码连接（pg_hba.conf限制本地连接） |
| 应用层认证 | SQLAlchemy连接池 + 环境变量管理密码 |
| API层认证 | JWT Token验证，管理员端点需is_admin=True |
| 密码存储 | bcrypt单向哈希（passlib[bcrypt]），永不明文存储 |

### 8.2 数据完整性

| 约束类型 | 实例 |
|---------|------|
| 实体完整性 | 所有表主键自增 |
| 参照完整性 | 所有外键ON DELETE CASCADE |
| 域完整性 | NOT NULL、DEFAULT值、唯一约束 |
| 用户定义完整性 | status枚举约束(draft/confirmed/completed)、评分范围(0—5) |

### 8.3 SQL注入防护

所有数据库操作通过SQLAlchemy ORM执行，使用参数化查询。不使用字符串拼接SQL。GeoAlchemy2的GIS函数调用也通过ORM层的func包装。

### 8.4 备份策略

- 课设场景：pg_dump手动导出
- 生产扩展：pg_dump定时任务 + WAL归档
- 导出命令：`pg_dump -U travel_planner travel_planner > backup.sql`
