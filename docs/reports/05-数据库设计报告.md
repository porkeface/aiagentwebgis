## 1. 引言

### 1.1 编写目的

本报告旨在对"基于AI Agent的旅游路线智能规划与景点空间推荐系统"的数据库进行系统化设计，详细阐述从需求分析、概念结构设计、逻辑结构设计到物理结构设计的全过程。本文档作为课程设计的核心技术文档，为后续数据库实施、应用程序开发及系统维护提供规范化的技术依据。

### 1.2 项目背景

随着旅游行业的数字化发展和人工智能技术的广泛应用，旅游路线智能规划成为智慧旅游领域的核心需求。本系统利用AI Agent技术，结合空间数据库（PostGIS）的地理分析能力，为用户提供智能化的旅游路线规划与景点空间推荐服务。

系统需要管理大量的兴趣点（POI）空间数据，支持复杂的空间查询与分析操作，同时需要管理用户信息、行程规划、对话会话等业务数据。选择PostgreSQL作为关系型数据库管理系统，并扩展PostGIS空间数据库引擎，以满足空间数据存储、索引和查询的需求。

### 1.3 定义与缩略语

| 术语 | 说明 |
|------|------|
| POI | Point of Interest，兴趣点，泛指景点、餐厅、酒店等地理实体 |
| PostGIS | PostgreSQL的空间数据库扩展，提供地理对象存储和空间分析能力 |
| GIST | Generalized Search Tree，PostgreSQL的通用索引结构，用于空间数据索引 |
| GIN | Generalized Inverted Index，广义倒排索引，适用于数组和JSONB类型 |
| JSONB | PostgreSQL的二进制JSON数据类型，支持高效查询和索引 |
| UUID | Universally Unique Identifier，通用唯一标识符 |
| E-R图 | Entity-Relationship Diagram，实体-关系图 |
| 3NF | Third Normal Form，第三范式 |
| SRID | Spatial Reference Identifier，空间参考标识符，本系统使用4326（WGS84） |
| Agent | AI智能代理，负责与用户对话、生成行程规划方案 |

## 2. 需求分析

### 2.1 数据需求

根据系统功能模块划分，识别出以下核心数据实体及其属性需求：

**（1）POI（兴趣点）实体**

POI是系统的核心数据资产，承载景点、餐厅、酒店、交通枢纽等地理实体信息。其属性包括：

- 基本信息：名称、分类（scenic/restaurant/hotel/transport/shopping）、子分类、所在城市、详细地址、描述说明
- 空间信息：经纬度坐标（SRID:4326）、几何点（GEOMETRY类型）
- 属性信息：综合评分（0-5）、价格等级（1-5）、标签数组、扩展数据（JSONB格式存储灵活属性）
- 来源信息：数据来源标识（amap：高德地图/ osm：OpenStreetMap / seed：种子数据）

**（2）用户（users）实体**

系统的注册使用者，属性包括：

- 基本信息：用户名（唯一）、邮箱（唯一）、密码哈希值
- 旅行偏好：偏好分类列表、预算等级（budget/mid/luxury）、旅行节奏（relaxed/moderate/intense），以JSONB格式存储
- 时间戳：创建时间、更新时间

**（3）行程（trip）实体**

由Agent为用户生成的旅行计划，属性包括：

- 关联用户（外键）
- 行程标题、目标城市、起止日期
- 状态标识（draft/confirmed/completed）
- Agent生成的完整行程方案（JSONB格式）
- 时间戳

**（4）行程日计划（trip_day）实体**

行程下按天拆分的日程安排，属性包括：

- 关联行程（外键）
- 天数序号（第1天、第2天...）、具体日期、备注信息

**（5）行程景点关联（trip_day_poi）实体**

日计划中具体景点的访问安排，属性包括：

- 关联日计划（外键）
- 关联POI（外键）
- 游览顺序、预计到达时间、停留时长（分钟）、交通方式（walking/bus/metro/taxi）

**（6）对话会话（chat_session）实体**

用户与Agent的对话记录，属性包括：

- 关联用户（外键）
- 会话ID（UUID，唯一）
- 消息列表（JSONB数组格式，包含角色、内容、时间戳）
- Agent上下文状态（JSONB格式，存储当前规划上下文）
- 时间戳

### 2.2 数据处理需求

系统核心查询与操作需求如下：

| 编号 | 查询类型 | 描述 | 涉及技术 |
|------|----------|------|----------|
| Q1 | 空间范围查询 | 查询指定坐标点周围N公里内的POI | ST_DWithin + GIST索引 |
| Q2 | 空间区域查询 | 查询指定矩形区域内的POI | ST_MakeEnvelope + ST_Within |
| Q3 | 标签搜索 | 按标签数组筛选POI（如"历史""美食"） | GIN索引 on TEXT[] |
| Q4 | JSONB查询 | 查询POI扩展属性、用户偏好 | JSONB操作符 + GIN索引 |
| Q5 | 行程关联查询 | 查询用户行程→日计划→景点的完整层级 | 多表JOIN |
| Q6 | 对话历史查询 | 按UUID查询会话消息 | UUID索引 |
| Q7 | 空间排序 | 按距离排序POI | ST_Distance + ORDER BY |
| Q8 | 聚合统计 | 按分类统计POI数量、平均评分 | GROUP BY + 聚合函数 |

### 2.3 数据量估算

| 数据表 | 预估数据量 | 增长趋势 | 说明 |
|--------|-----------|----------|------|
| poi | 5,000 ~ 10,000条 | 中速增长 | 单城市500-2000条，覆盖多个城市 |
| users | < 100条 | 缓慢增长 | 课程设计规模，少量测试用户 |
| trip | < 500条 | 中速增长 | 用户创建的行程计划 |
| trip_day | < 2,000条 | 与行程同步增长 | 每个行程含3-7天 |
| trip_day_poi | < 10,000条 | 快速增长 | 每天含3-8个景点 |
| chat_session | < 2,000条 | 中速增长 | 用户与Agent的对话会话 |

综合来看，系统总数据量在万级规模，属于中小型应用。PostgreSQL完全能够支撑该数据量的高效运行。

## 4. 逻辑结构设计

### 4.1 关系模式设计

根据E-R图向关系模式的转换规则：

1. **实体转换**：每个实体转换为一个关系模式，实体属性成为关系属性，实体主键成为关系主键。
2. **1:N关系转换**：在"N"端关系中引入"1"端关系的主键作为外键。
3. **M:N关系转换**：本系统中trip_day与poi之间为M:N关系，通过trip_day_poi关联表（已含附加属性order_num、arrive_time等）自然转换。

### 4.2 关系模式规范化分析

**第一范式（1NF）分析：**

所有关系模式中的属性均为原子值或明确定义的组合类型（GEOMETRY、JSONB、TEXT[]）。PostgreSQL的复合类型为数据库原生支持的一级数据类型，具有明确的语义和操作语义，满足1NF要求。

**第二范式（2NF）分析：**

所有非主属性完全函数依赖于主键，不存在部分函数依赖：

- `poi(id)` → name, category, ..., location：所有属性依赖于id
- `users(id)` → username, email, ..., preferences：所有属性依赖于id
- `trip(id)` → user_id, title, ..., plan_data：所有属性依赖于id
- `trip_day(id)` → trip_id, day_number, date, notes：所有属性依赖于id
- `trip_day_poi(id)` → trip_day_id, poi_id, order_num, ...：所有属性依赖于id
- `chat_session(id)` → user_id, session_id, messages, ...：所有属性依赖于id

各关系均满足2NF。

**第三范式（3NF）分析：**

检查是否存在非主属性对主键的传递依赖：

- `poi`表：各属性直接描述POI本身，无传递依赖 ✓
- `users`表：各属性直接描述用户本身，无传递依赖 ✓
- `trip`表：user_id为外键，其余属性直接描述行程，无传递依赖 ✓
- `trip_day`表：trip_id为外键，其余属性直接描述日计划，无传递依赖 ✓
- `trip_day_poi`表：trip_day_id和poi_id均为外键，其余属性直接描述访问安排，无传递依赖 ✓
- `chat_session`表：user_id为外键，其余属性直接描述会话，无传递依赖 ✓

各关系均满足3NF，无需进一步分解。

### 4.3 完整的关系模式列表

```
POI (id, name, category, sub_category, city, address, description,
     location, rating, price_level, tags, extra_data, source,
     created_at, updated_at)

users (id, username, email, password_hash, preferences,
       created_at, updated_at)

trip (id, user_id, title, city, start_date, end_date, status,
      plan_data, created_at, updated_at)

trip_day (id, trip_id, day_number, date, notes)

trip_day_poi (id, trip_day_id, poi_id, order_num, arrive_time,
              duration_min, transport_mode)

chat_session (id, user_id, session_id, messages, agent_context,
              created_at, updated_at)
```

## 5. 物理结构设计

### 5.1 完整的建表SQL语句

```sql
-- ============================================================
-- 数据库：travel_planner
-- 依赖扩展：postgis
-- ============================================================

-- 启用PostGIS扩展（如未启用）
CREATE EXTENSION IF NOT EXISTS postgis;

-- ============================================================
-- 1. POI（兴趣点）表
-- ============================================================
CREATE TABLE poi (
    id          BIGSERIAL       PRIMARY KEY,
    name        VARCHAR(200)    NOT NULL,
    category    VARCHAR(50)     NOT NULL,
    sub_category VARCHAR(100),
    city        VARCHAR(100)    NOT NULL,
    address     VARCHAR(500),
    description TEXT,
    location    GEOMETRY(Point, 4326) NOT NULL,
    rating      NUMERIC(2,1)    CHECK (rating >= 0 AND rating <= 5),
    price_level SMALLINT        CHECK (price_level >= 1 AND price_level <= 5),
    tags        TEXT[],
    extra_data  JSONB           DEFAULT '{}'::jsonb,
    source      VARCHAR(50)     NOT NULL DEFAULT 'seed'
                                CHECK (source IN ('amap', 'osm', 'seed')),
    created_at  TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ     NOT NULL DEFAULT NOW()
);

COMMENT ON TABLE  poi IS '兴趣点表：存储景点、餐厅、酒店等地理实体';
COMMENT ON COLUMN poi.id IS 'POI唯一标识（自增主键）';
COMMENT ON COLUMN poi.name IS 'POI名称';
COMMENT ON COLUMN poi.category IS '分类：scenic/restaurant/hotel/transport/shopping';
COMMENT ON COLUMN poi.sub_category IS '子分类：如景点下的"历史遗迹""自然风景"等';
COMMENT ON COLUMN poi.city IS '所在城市';
COMMENT ON COLUMN poi.address IS '详细地址';
COMMENT ON COLUMN poi.description IS 'POI描述信息';
COMMENT ON COLUMN poi.location IS '空间坐标点（WGS84, SRID:4326）';
COMMENT ON COLUMN poi.rating IS '综合评分，范围0.0-5.0';
COMMENT ON COLUMN poi.price_level IS '价格等级，1=经济 2=适中 3=中等 4=偏贵 5=豪华';
COMMENT ON COLUMN poi.tags IS '标签数组，用于灵活分类和检索';
COMMENT ON COLUMN poi.extra_data IS '扩展属性（JSONB），存储灵活的业务字段';
COMMENT ON COLUMN poi.source IS '数据来源：amap=高德地图, osm=OpenStreetMap, seed=种子数据';

-- ============================================================
-- 2. 用户表
-- ============================================================
CREATE TABLE users (
    id            BIGSERIAL       PRIMARY KEY,
    username      VARCHAR(100)    NOT NULL UNIQUE,
    email         VARCHAR(200)    NOT NULL UNIQUE,
    password_hash VARCHAR(255)    NOT NULL,
    preferences   JSONB           DEFAULT '{}'::jsonb,
    created_at    TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at    TIMESTAMPTZ     NOT NULL DEFAULT NOW()
);

COMMENT ON TABLE  users IS '用户表：存储注册用户的账户信息和旅行偏好';
COMMENT ON COLUMN users.id IS '用户唯一标识（自增主键）';
COMMENT ON COLUMN users.username IS '用户名（唯一）';
COMMENT ON COLUMN users.email IS '邮箱地址（唯一）';
COMMENT ON COLUMN users.password_hash IS '密码哈希值（bcrypt/argon2）';
COMMENT ON COLUMN users.preferences IS '旅行偏好（JSONB），含preferred_categories/budget_level/travel_pace';

-- ============================================================
-- 3. 行程表
-- ============================================================
CREATE TABLE trip (
    id          BIGSERIAL       PRIMARY KEY,
    user_id     BIGINT          NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    title       VARCHAR(200)    NOT NULL,
    city        VARCHAR(100)    NOT NULL,
    start_date  DATE            NOT NULL,
    end_date    DATE            NOT NULL,
    status      VARCHAR(20)     NOT NULL DEFAULT 'draft'
                                CHECK (status IN ('draft', 'confirmed', 'completed')),
    plan_data   JSONB           DEFAULT '{}'::jsonb,
    created_at  TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    CONSTRAINT chk_date_range CHECK (end_date >= start_date)
);

COMMENT ON TABLE  trip IS '行程表：存储用户创建的旅行计划';
COMMENT ON COLUMN trip.id IS '行程唯一标识（自增主键）';
COMMENT ON COLUMN trip.user_id IS '创建者用户ID（外键→users）';
COMMENT ON COLUMN trip.title IS '行程标题';
COMMENT ON COLUMN trip.city IS '目标城市';
COMMENT ON COLUMN trip.start_date IS '行程起始日期';
COMMENT ON COLUMN trip.end_date IS '行程结束日期';
COMMENT ON COLUMN trip.status IS '行程状态：draft=草稿, confirmed=已确认, completed=已完成';
COMMENT ON COLUMN trip.plan_data IS 'Agent生成的完整行程方案（JSONB）';

-- ============================================================
-- 4. 行程日计划表
-- ============================================================
CREATE TABLE trip_day (
    id          BIGSERIAL       PRIMARY KEY,
    trip_id     BIGINT          NOT NULL REFERENCES trip(id) ON DELETE CASCADE,
    day_number  SMALLINT        NOT NULL CHECK (day_number > 0),
    date        DATE            NOT NULL,
    notes       TEXT,
    CONSTRAINT uq_trip_day UNIQUE (trip_id, day_number)
);

COMMENT ON TABLE  trip_day IS '行程日计划表：行程按天拆分的日程安排';
COMMENT ON COLUMN trip_day.id IS '日计划唯一标识（自增主键）';
```

### 5.3 分区策略

鉴于本系统数据量在万级规模（课设级别），暂不采用表分区策略。PostgreSQL在该数据量下通过合理的索引设计即可实现高效的查询性能。

若未来系统扩展至百万级以上数据量，可考虑以下分区方案：

- **poi表**：按`city`字段进行LIST分区，每个城市一个分区
- **chat_session表**：按`created_at`进行RANGE分区，按月份划分历史数据
- **trip表**：按`status`进行LIST分区，区分活跃行程与历史行程

## 7. 数据完整性设计

### 7.1 实体完整性（主键约束）

所有表均采用`BIGSERIAL`自增类型作为主键（`id`列），保证每条记录的唯一标识。主键约束隐含`NOT NULL`和`UNIQUE`语义，由PostgreSQL自动创建B-tree唯一索引。

| 表名 | 主键 | 索引类型 |
|------|------|---------|
| poi | id | B-tree（自动） |
| users | id | B-tree（自动） |
| trip | id | B-tree（自动） |
| trip_day | id | B-tree（自动） |
| trip_day_poi | id | B-tree（自动） |
| chat_session | id | B-tree（自动） |

### 7.2 参照完整性（外键约束）

| 外键表 | 外键列 | 引用表 | 引用列 | 删除策略 | 说明 |
|--------|--------|--------|--------|---------|------|
| trip | user_id | users | id | CASCADE | 用户删除→行程删除 |
| trip_day | trip_id | trip | id | CASCADE | 行程删除→日计划删除 |
| trip_day_poi | trip_day_id | trip_day | id | CASCADE | 日计划删除→景点安排删除 |
| trip_day_poi | poi_id | poi | id | RESTRICT | POI被引用时禁止删除 |
| chat_session | user_id | users | id | CASCADE | 用户删除→会话删除 |

**设计说明**：

- 采用`ON DELETE CASCADE`的关系表示强依赖关系（如日计划依赖于行程存在）。
- 采用`ON DELETE RESTRICT`的关系表示弱依赖关系（POI作为基础数据，被行程引用时不应被删除）。

### 7.3 域完整性（CHECK约束与NOT NULL）

| 约束名 | 表 | 约束表达式 | 说明 |
|--------|-----|-----------|------|
| chk_poi_rating | poi | rating >= 0 AND rating <= 5 | 评分范围限制 |
| chk_poi_price | poi | price_level >= 1 AND price_level <= 5 | 价格等级范围 |
| chk_poi_source | poi | source IN ('amap', 'osm', 'seed') | 数据来源枚举 |
| chk_trip_status | trip | status IN ('draft', 'confirmed', 'completed') | 行程状态枚举 |
| chk_trip_dates | trip | end_date >= start_date | 日期范围合理性 |
| chk_day_number | trip_day | day_number > 0 | 天数序号为正整数 |
| chk_order_num | trip_day_poi | order_num > 0 | 游览顺序为正整数 |
| chk_duration | trip_day_poi | duration_min > 0 OR duration_min IS NULL | 停留时长为正 |
| chk_transport | trip_day_poi | transport_mode IN ('walking','bus','metro','taxi','driving') OR NULL | 交通方式枚举 |

### 7.4 用户自定义完整性

- **唯一约束**：`users.username`和`users.email`保证用户标识唯一性；`trip_day(trip_id, day_number)`和`trip_day_poi(trip_day_id, order_num)`保证排序字段不重复。
- **触发器（可选扩展）**：可通过触发器自动更新`updated_at`字段：

```sql
CREATE OR REPLACE FUNCTION update_timestamp()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_poi_updated BEFORE UPDATE ON poi
    FOR EACH ROW EXECUTE FUNCTION update_timestamp();

CREATE TRIGGER trg_users_updated BEFORE UPDATE ON users
    FOR EACH ROW EXECUTE FUNCTION update_timestamp();

CREATE TRIGGER trg_trip_updated BEFORE UPDATE ON trip
    FOR EACH ROW EXECUTE FUNCTION update_timestamp();

CREATE TRIGGER trg_chat_updated BEFORE UPDATE ON chat_session
    FOR EACH ROW EXECUTE FUNCTION update_timestamp();
```

## 10. 核心SQL查询示例

### 10.1 空间范围查询

**查询指定坐标点3公里范围内的景点：**

```sql
SELECT id, name, category, rating,
       ST_Distance(
           location::geography,
           ST_SetSRID(ST_MakePoint(116.397, 39.908), 4326)::geography
       ) AS distance_meters
FROM poi
WHERE ST_DWithin(
    location::geography,
    ST_SetSRID(ST_MakePoint(116.397, 39.908), 4326)::geography,
    3000  -- 3000米
)
AND category = 'scenic'
ORDER BY distance_meters ASC
LIMIT 20;
```

### 10.2 景点搜索查询

**按标签和评分筛选POI：**

```sql
SELECT id, name, category, rating, tags
FROM poi
WHERE city = '北京'
  AND category = 'restaurant'
  AND tags @> ARRAY['火锅', '老字号']  -- 包含指定标签
  AND rating >= 4.0
  AND price_level <= 3
ORDER BY rating DESC
LIMIT 20;
```

**JSONB扩展数据查询：**

```sql
SELECT id, name, extra_data->>'opening_hours' AS hours,
       extra_data->>'phone' AS phone
FROM poi
WHERE city = '上海'
  AND extra_data ? 'opening_hours'  -- 含营业时间字段
  AND extra_data @> '{"wheelchair_accessible": true}'::jsonb;
```

### 10.3 行程关联查询

**查询用户行程的完整日计划及景点信息：**

```sql
SELECT
    t.id AS trip_id,
    t.title AS trip_title,
    t.city AS trip_city,
    t.status,
    td.day_number,
    td.date,
    tdp.order_num,
    p.name AS poi_name,
    p.category,
    p.rating AS poi_rating,
    tdp.arrive_time,
    tdp.duration_min,
    tdp.transport_mode
FROM trip t
JOIN trip_day td ON td.trip_id = t.id
JOIN trip_day_poi tdp ON tdp.trip_day_id = td.id
JOIN poi p ON p.id = tdp.poi_id
WHERE t.user_id = 1
  AND t.status = 'confirmed'
ORDER BY td.day_number, tdp.order_num;
```

### 10.4 对话历史查询

**按UUID查询会话记录：**

```sql
SELECT session_id, user_id, messages, agent_context, created_at, updated_at
FROM chat_session
WHERE session_id = 'a1b2c3d4-e5f6-7890-abcd-ef1234567890';
```

**查询用户最近10个会话：**

```sql
SELECT session_id, 
       messages->0->>'content' AS first_message,
       jsonb_array_length(messages) AS message_count,
       created_at
FROM chat_session
WHERE user_id = 1
ORDER BY updated_at DESC
LIMIT 10;
```

### 10.5 空间聚合查询

**按分类统计城市内POI数量和平均评分：**

```sql
SELECT city, category,
       COUNT(*) AS poi_count,
       ROUND(AVG(rating), 2) AS avg_rating,
       ROUND(MIN(rating), 1) AS min_rating,
       ROUND(MAX(rating), 1) AS max_rating
FROM poi
WHERE city = '北京'
GROUP BY city, category
ORDER BY poi_count DESC;
```
