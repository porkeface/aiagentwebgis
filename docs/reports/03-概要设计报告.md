本报告是"基于AI Agent的旅游路线智能规划与景点空间推荐系统"课程设计的概要设计（总体设计）文档。编写目的在于：

### 1.2 项目背景

随着旅游业的快速复苏和人工智能技术的日益成熟，传统旅游路线规划方式——依赖人工查找攻略、手动标注地图——已无法满足用户对高效、智能、个性化出行方案的需求。与此同时，大语言模型（LLM）的推理能力与工具调用（Tool Calling）机制日趋成熟，为构建具有自主决策能力的智能代理（AI Agent）提供了技术基础。

本项目旨在设计并实现一个融合 AI Agent 推理能力与 WebGIS 空间分析能力的旅游路线智能规划系统。系统采用 LangGraph StateGraph 框架构建多节点 Agent 工作流，通过 Router → Planner → Formatter 的流水线架构，结合 ReAct 推理模式与多工具协同调用，实现从用户自然语言输入到完整行程方案输出的端到端智能规划。同时，系统内置基于 PostGIS 的空间推荐引擎，采用"空间过滤→多因子评分→MMR 多样性重排→DBSCAN 聚类→TSP 路线优化"五步流水线算法，确保推荐结果在空间分布上的合理性与游览体验的多样性。

## 2 总体设计

### 2.1 设计原则

本系统的架构设计遵循以下核心原则：

**（1）分层解耦原则。** 系统采用严格的四层架构，各层通过明确定义的接口进行通信。上层仅依赖下层的抽象接口，不感知其内部实现细节。当某一层的内部实现发生变化时，只要接口契约不变，其余各层无需修改。

**（2）关注点分离原则。** 每个模块仅负责单一领域的职责。前端专注于可视化渲染与用户交互；后端专注于业务逻辑与数据管理；Agent 层专注于推理决策；数据层专注于存储与缓存。避免模块间的职责交叉与功能耦合。

**（3）可扩展性原则。** Agent 工具链采用插件式注册机制，新增工具（如航班查询、酒店预订）仅需实现标准接口并注册至工具表，无需修改 Agent 核心逻辑。推荐引擎流水线各步骤可独立替换与升级。

**（4）渐进增强原则。** 系统采用"种子数据兜底 + 实时API查询"的数据策略，即使在外部API不可用的情况下，仍能保证基本功能运行。LLM适配器采用双路策略：优先使用原生 tool_call，异常时自动回退至 prompt-based 模拟调用。

**（5）实时反馈原则。** 通过 SSE（Server-Sent Events）流式通信协议，将 Agent 的推理过程（思考、工具调用、中间结果）实时推送至前端，避免用户在长时间推理过程中的"黑箱等待"焦虑。

#### 2.2.2 各层职责说明

**第一层：前端 WebGIS 可视化层。** 负责系统的用户界面呈现与交互逻辑。核心职责包括：（a）基于 Leaflet 的地图渲染与 POI 标注；（b）用户对话界面的消息展示与 PlanCard 渲染；（c）行程时间线的可视化展示；（d）通过 SSE 接收 Agent 推理过程的实时流式数据并驱动 UI 更新；（e）通过 Pinia 管理全局状态（地图视角、对话上下文、行程数据）。

**第二层：后端 API 服务层。** 负责业务逻辑处理与数据管理。核心职责包括：（a）提供 RESTful API 供前端进行数据查询与 CRUD 操作；（b）作为 Agent 的网关层，接收用户输入后启动 LangGraph 图执行流程；（c）通过 SSE 协议将 Agent 各节点的输出事件实时推送至前端；（d）JWT 认证与权限控制；（e）PostGIS 空间查询与推荐引擎计算。

**第三层：AI Agent 决策层。** 负责智能推理与任务编排。核心职责包括：（a）Router 节点对用户输入进行意图识别与分类；（b）Planner Agent 以 ReAct 模式进行多步推理，根据任务需要动态选择和调用工具；（c）Formatter 节点将 Agent 输出的结构化数据转换为前端可渲染的格式化方案；（d）通过 StateGraph 共享白板实现节点间的状态同步与协调；（e）支持回环机制——Route Optimizer 发现景点间距过远时可回到 Planner 重新选择。

**第四层：数据存储层。** 负责数据的持久化存储与高效检索。核心职责包括：（a）PostgreSQL + PostGIS 存储 POI 空间数据、用户数据、行程数据，支持空间索引与地理查询；（b）Redis 提供热点数据缓存、SSE 会话管理和 API 响应缓存；（c）高德地图 API 提供实时地理编码、路线规划、POI 搜索能力；（d）LLM API（通义千问/DashScope）提供自然语言理解与推理能力。

## 3 模块设计

### 3.1 前端模块设计

前端采用 Vue 3 + Vite + TypeScript 技术栈，按功能域划分为三大核心模块。

#### 3.1.1 地图展示模块

```

**职责：** 以 Leaflet 为底图渲染引擎，加载高德地图瓦片服务；通过 PoiMarker 组件标注 POI 兴趣点，支持点击查看详情、高亮选中状态；RouteLayer 组件绘制景点间路线，支持按天着色；MapControls 提供缩放、定位等地图操控功能。地图视角通过 useMapStore 实现响应式管理，支持根据 Agent 返回的 POI 数据自动调整视口范围。

**（1）认证模块。** 基于 JWT（JSON Web Token）实现无状态认证。用户注册时对密码进行 bcrypt 哈希加密后存储；登录时验证凭据并签发 Access Token；受保护的路由通过 Depends 依赖注入方式验证 Token 有效性。

**（2）POI 服务模块。** 提供 POI 数据的增删改查与空间查询能力。空间查询基于 PostGIS 的 ST_DWithin、ST_MakeEnvelope 等函数实现区域过滤；同时封装高德 POI 搜索 API 作为实时数据补充。种子数据管理组件在数据库为空时自动导入预置景点数据，确保系统基础功能的可用性。

**（3）行程管理模块。** 负责行程方案的持久化管理，支持行程的创建、编辑、删除与查询。行程数据与 POI 通过外键关联，支持多日日程的嵌套结构。路线数据以 GeoJSON 格式存储于 PostGIS 的 Geometry 列中。

**（4）Agent 网关模块。** 作为前端与 Agent 层的桥梁，接收用户输入后启动 LangGraph 图执行流程。核心职责是将 Agent 各节点的输出事件转化为统一的 SSE 消息体，通过 event/data 字段对推送至前端。消息体采用 type 字段分发机制，定义 thinking、tool_calling、poi_result、route_result、plan_summary、text、error 七种消息类型。同时管理 Checkpoint 状态持久化，支持对话会话的断点续传。

### 3.3 Agent 模块设计

Agent 模块是本系统的核心智能层，采用 LangGraph StateGraph 实现基于共享白板的多节点协作架构。

#### 3.3.1 整体架构

```
                         用户输入
                            │
                            ▼
                    ┌───────────────┐
                    │    Router     │
                    │  (条件路由节点) │
                    └───────┬───────┘
                            │
              ┌─────────────┼─────────────┐
              │             │             │
              ▼             ▼             ▼
      ┌──────────┐  ┌──────────┐  ┌──────────┐
      │ 路线规划  │  │ POI查询  │  │ 闲聊应答  │
      │ 意图分支  │  │ 意图分支  │  │ 意图分支  │
      └────┬─────┘  └────┬─────┘  └────┬─────┘
           │             │             │
           ▼             │             │
  ┌────────────────┐     │             │
  │ Planner Agent  │◄────┘             │
  │ (ReAct + Tools)│                    │
  └────────┬───────┘                    │
           │                            │
     ┌─────┴─────┐                      │
     │           │                      │
     ▼           ▼                      │
┌─────────┐ ┌──────────┐               │

│ (格式化) │ │ Optimizer│               │

     │      └────┬─────┘               │
     │           │                      │
     │     (景点过远?回环)──────────────┘
     │
     ▼
  最终输出 (SSE → 前端)
```

#### 3.3.2 各节点设计

**Router（条件路由节点）。** 作为 Agent 工作流的入口，负责解析用户输入并判断意图类别。Router 采用条件路由机制：通过 LLM 对用户输入进行意图分类，将请求路由至相应的处理分支。意图类别包括但不限于：路线规划请求、POI 信息查询、一般性闲聊等。Router 节点不做具体业务处理，仅负责分类与路由决策。

**Planner Agent（核心推理节点）。** 作为整个 Agent 系统的"大脑"，Planner 采用 ReAct（Reasoning + Acting）模式进行多步推理。在每个推理步骤中，Planner 首先分析当前状态（State），决定下一步应采取的行动（调用哪个工具），然后根据工具返回的结果继续推理。Planner 可调用多种工具完成复杂任务，工具之间通过共享 State 白板传递数据。

**Formatter（格式化输出节点）。** 将 Planner 输出的结构化数据（包含景点列表、日程安排、路线信息等）转换为前端可直接渲染的格式化方案。Formatter 负责数据的最终组装与呈现格式的决定，确保输出数据符合前端 PlanCard 组件的渲染规范。

**Route Optimizer（路线优化节点）。** 负责对 Planner 选定的景点进行空间路线优化。若发现景点间距离过远或路线不合理，Route Optimizer 可触发回环机制——修改 State 中的 selected_pois 字段后，将控制权交还 Planner 重新选择景点，从而实现"规划→验证→修正"的闭环优化。

#### 3.3.3 工具链设计

```
┌─────────────────────────────────────────────────┐
│                  Agent 工具链                    │
├──────────────┬──────────────┬───────────────────┤
│  POI 搜索工具 │  地理编码工具  │   路线规划工具     │
│  poi_search  │  geocoding   │   route_planning   │
│              │              │                    │
│  ·关键字搜索  │  ·地址→坐标   │   ·驾车路线        │

│  ·区域过滤   │  ·城市编码    │   ·公交路线        │

工具链采用插件式注册机制。每个工具实现 LangChain 的 StructuredTool 接口，包含 name、description、args_schema 三个核心属性。工具通过 LangGraph 的 ToolNode 注册至 Planner Agent，Planner 在推理过程中根据任务需要动态选择并调用工具。新增工具仅需实现标准接口并注册，无需修改 Agent 核心逻辑。

Redis 缓存层采用 Cache-Aside 模式：后端服务先查询缓存，命中则直接返回；未命中则查询数据库或外部API，获取结果后写入缓存并设置合理的 TTL（过期时间）。城市级 POI 缓存 1 小时，地理编码缓存 24 小时，路线数据缓存 2 小时，天气数据缓存 30 分钟。Agent 会话状态通过 LangGraph Checkpoint 机制持久化至 Redis，支持断点续传。

#### 4.1.2 SSE 流式接口

Agent 规划接口（`POST /api/v1/agent/plan`）采用 SSE（Server-Sent Events）协议进行流式响应推送。

**请求体：**

```json
{
  "message": "帮我规划一个杭州三日游",
  "session_id": "可选，用于断点续传"
}
```

**SSE 消息协议：**

```
event: message
data: {"type": "thinking", "content": "正在分析您的旅行需求...", "timestamp": 1719600000}

event: message
data: {"type": "tool_calling", "content": {"tool": "poi_search", "args": {"city": "杭州"}}, "timestamp": 1719600001}

event: message
data: {"type": "poi_result", "content": {"pois": [...]}, "timestamp": 1719600002}

event: message
data: {"type": "route_result", "content": {"routes": [...]}, "timestamp": 1719600003}

event: message
data: {"type": "plan_summary", "content": {"title": "杭州三日游", "days": 3}, "timestamp": 1719600004}

event: message
data: {"type": "text", "content": "已为您生成杭州三日游方案", "timestamp": 1719600005}

event: done
data: {}
```

**七种消息类型定义：**

| 类型           | 含义                       | 前端渲染方式               |
| -------------- | -------------------------- | -------------------------- |
| thinking       | Agent 正在思考和推理        | 折叠面板展示思考过程        |
| tool_calling   | Agent 正在调用工具          | 工具调用状态动画            |
| poi_result     | POI 搜索结果               | 地图上标注 POI 标记         |
| route_result   | 路线规划结果               | 地图上绘制路线             |
| plan_summary   | 行程方案摘要               | PlanCard 卡片渲染           |
| text           | Agent 的自然语言文本回复    | 消息气泡展示               |
| error          | 错误信息                   | 错误提示框展示              |

### 4.2 外部接口设计

#### 4.2.1 高德地图 API

| 接口             | 用途            | 调用时机                      |
| ---------------- | --------------- | ----------------------------- |
| POI 搜索         | 获取兴趣点数据  | Agent 工具 poi_search 调用时  |
| 地理编码         | 地名→坐标转换   | Agent 工具 geocoding 调用时   |
| 逆地理编码       | 坐标→地名转换   | 定位功能                      |
| 路线规划         | 计算最优路线    | Agent 工具 route_planning 调用|
| 静态地图         | 生成地图缩略图  | 行程分享                      |

调用策略：所有外部 API 调用经过 Redis 缓存层，相同参数的请求在 TTL 内直接返回缓存结果，减少外部 API 的调用频次与响应延迟。

#### 4.2.2 LLM API（通义千问/DashScope）

| 接口             | 模型            | 用途                          |
| ---------------- | --------------- | ----------------------------- |
| 对话补全         | qwen-max        | Agent 推理与意图识别          |
| 工具调用         | qwen-max        | Planner Agent 工具调用        |
| Embedding        | text-embedding  | POI 语义搜索（可扩展）        |

LLM 适配层采用适配器模式（Adapter Pattern），将 DashScope SDK 的接口统一转化为 LangChain 的 BaseChatModel 接口。适配器内部实现双路策略：优先尝试使用原生 tool_call 能力；当 SDK 返回格式异常或调用失败时，自动回退至 prompt-based 模拟工具调用（通过在 system prompt 中定义工具格式，解析 LLM 输出中的 JSON 工具调用指令）。

### 4.3 模块间接口设计

#### 4.3.1 前端 ↔ 后端接口

```
┌──────────────┐    REST (JSON)     ┌──────────────┐
│   前端 SPA   │ ──────────────────→ │  FastAPI     │
│              │ ←────────────────── │  Backend     │
│              │    SSE (EventStream)│              │
└──────────────┘                    └──────────────┘
```

通信协议：REST 请求使用 JSON 格式，Content-Type 为 application/json。SSE 连接使用 text/event-stream 格式。所有 API 响应采用统一的信封结构：`{"code": int, "message": str, "data": any}`。

#### 4.3.2 后端 ↔ Agent 接口

```
┌──────────────┐   函数调用     ┌──────────────┐
│  Agent 网关  │ ─────────────→ │ LangGraph    │
│  (FastAPI)   │ ←───────────── │ StateGraph   │
│              │   State 事件流  │              │
└──────────────┘               └──────────────┘
```

Agent 网关通过 LangGraph 的 `astream_events` 方法获取图的执行事件流，将每个节点的状态变化转化为 SSE 消息推送至前端。Agent 的输入为包含用户消息和会话 ID 的字典，输出为共享 State 中的结构化行程数据。

#### 4.3.3 Agent ↔ 工具链接口

```
┌──────────────┐   Tool 调用     ┌──────────────┐
│   Planner    │ ─────────────→ │  Tools       │
│   Agent      │ ←───────────── │  Registry    │
│              │   Tool 返回值   │              │
└──────────────┘               └──────────────┘
```

工具调用遵循 LangChain StructuredTool 接口规范。Planner 通过 LLM 的 tool_call 机制发起工具调用请求，工具执行后将结果写入共享 State 的对应字段。

### 5.3 推荐引擎数据流

推荐引擎采用五步流水线架构，从原始 POI 数据中筛选并优化出最终推荐的景点集合与游览路线。

```
┌──────────────────────────────────────────────────────────────────────┐
│                    推荐引擎五步流水线                                   │
│                                                                      │
│  Step 1: 空间过滤                                                     │
│  ┌────────────────────────────────────────────────────────────────┐  │
│  │  输入: 目标城市 + 用户偏好 + 空间范围                            │  │
│  │  方法: PostGIS ST_DWithin / ST_MakeEnvelope                    │  │
│  │  输出: 候选 POI 集合 (candidate_pois)                          │  │
│  │  过滤条件: 距离约束 | 类别匹配 | 评分阈值                       │  │
│  └───────────────────────────────┬────────────────────────────────┘  │
│                                  │                                   │
│                                  ▼                                   │
│  Step 2: 多因子评分                                                    │
│  ┌────────────────────────────────────────────────────────────────┐  │
│  │  评分因子: 用户评分(0.3) + 热度(0.2) + 距离(0.2) + 类别匹配(0.3)│  │
│  │  公式: score = Σ(wi × fi)                                      │  │
│  │  输出: 带评分排序的 POI 列表                                     │  │
│  └───────────────────────────────┬────────────────────────────────┘  │
│                                  │                                   │
│                                  ▼                                   │
│  Step 3: MMR 多样性重排                                                │
│  ┌────────────────────────────────────────────────────────────────┐  │
│  │  目标: 平衡相关性与多样性，避免推荐同质化景点                      │  │
│  │  参数: λ=0.7 (相关性权重)                                       │  │
│  │  公式: MMR = argmax[λ·Relevance(d) - (1-λ)·max_sim(d, d')]     │  │
│  │  输出: 多样性排序的 POI 列表                                     │  │
│  └───────────────────────────────┬────────────────────────────────┘  │
│                                  │                                   │
│                                  ▼                                   │
│  Step 4: DBSCAN 空间聚类                                               │
│  ┌────────────────────────────────────────────────────────────────┐  │
│  │  目标: 将空间邻近的 POI 分组，便于按区域分配至不同日期              │  │
│  │  参数: eps=2km, min_samples=3                                   │  │
│  │  输出: 聚类标签 (cluster_id → POI 列表)                         │  │
│  └───────────────────────────────┬────────────────────────────────┘  │
│                                  │                                   │
│                                  ▼                                   │
│  Step 5: TSP 路线优化                                                  │
│  ┌────────────────────────────────────────────────────────────────┐  │
│  │  目标: 为每个聚类（每日行程）计算最优游览顺序                      │  │
│  │  方法: 最近邻启发式 + 2-opt 局部优化                             │  │
│  │  距离矩阵: 基于高德路线规划 API 的实际路程距离                    │  │
│  │  输出: 每日有序景点列表 + 路线几何数据                            │  │
│  └────────────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────────────┘
```

### 6.5 架构模式选型论证

**（1）StateGraph 共享白板模式 vs 管道链式模式。** StateGraph 各节点通过共享 State 白板读写数据，任意节点可读取全局上下文，支持回环（Route Optimizer→Planner）。管道链式模式数据单向流动，不支持回环，不适合需要"验证-修正"闭环的规划任务。

**（2）SSE vs WebSocket。** SSE 是单向服务器推送，基于 HTTP 协议，天然支持断线重连与负载均衡，与 REST API 共用端口。WebSocket 是双向通信，实现复杂度更高，但本项目仅需服务器向客户端推送，SSE 已完全满足需求且更轻量。

**（3）Cache-Aside vs Write-Through。** Cache-Aside 模式实现简单，读时查缓存、未命中则查库并回填，适合读多写少的旅游 POI 场景。Write-Through 模式在写入时同步更新缓存，增加写入开销，但本项目 POI 数据变更频率低，Cache-Aside 更为合适。
