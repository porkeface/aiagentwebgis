## 1 引言

### 1.1 编写目的

本报告完成系统的总体架构设计，明确四层分层架构的边界与交互关系，定义模块划分与职责，设计内外接口规范，阐述核心数据流与Agent状态流转逻辑。

### 1.2 项目背景

系统采用 LangGraph StateGraph 构建Agent工作流，以FastAPI为API服务层，以Vue 3 + 高德JS API为前端可视化层，以PostgreSQL + PostGIS为空间数据引擎。核心技术特色在于**确定性规划流水线**——意图分类后不走LLM逐步推理，而是进入一个搜索→评分→分区→路径→验证的十步流水线，避免了LLM规划中的幻觉问题。

### 1.3 定义与缩略语

| 术语 | 含义 |
|------|------|
| AI Agent | 基于LLM的智能代理，具备工具调用与多步推理能力 |
| LangGraph | 基于图结构的LLM Agent编排框架 |
| SSE | Server-Sent Events，服务器推送事件 |
| POI | Point of Interest，兴趣点 |
| PostGIS | PostgreSQL空间数据库扩展 |
| ReAct | Reasoning + Acting，推理-行动模式 |
| DBSCAN | 基于密度的空间聚类算法 |
| Checkpoint | LangGraph状态检查点，用于多轮对话状态恢复 |

## 2 总体设计

### 2.1 设计原则

**(1) 分层解耦原则。** 四层架构，层间通过明确定义的接口通信，上层不感知下层实现细节。

**(2) 确定性优先原则。** 行程规划不依赖LLM逐步工具调用，而是用确定性的流水线算法。LLM仅用于意图分类和最终摘要，规划质量稳定可复现。

**(3) 可扩展性原则。** Agent工具链采用插件式注册，LLM通过ChatOpenAI统一接口适配多供应商，配置白名单支持运行时热重载。

**(4) 实时反馈原则。** SSE流式推送Agent推理全过程（20种事件类型），用户实时感知进度。

### 2.3 各层职责

**前端可视化层：** 高德地图渲染与POI标注（RouteLayer自定义渲染器）、对话消息展示与SSE流接收、Pinia状态管理、管理后台配置界面。

**API服务层：** RESTful API（认证/行程/POI/管理）、SSE流式推送、JWT认证、滑动窗口速率限制、配置热重载。

**Agent决策层：** 意图分类（4类意图）、Agent ReAct循环（chat/poi_query/route_query）、确定性规划流水线（trip_planning）、LLM摘要生成、7个工具的统一调度、AmapService单例管理。

**数据存储层：** PostgreSQL+PostGIS空间数据存储（POI/用户/行程/会话）、高德API（地理编码/POI搜索/路径规划）、LLM API（DashScope默认，可切换DeepSeek）。

### 2.4 Agent工作流

```
用户消息 → classify_intent_node (LLM分类)
              │
    ┌─────────┼─────────┐
    │         │         │
  chat    poi_query  trip_planning
    │         │         │
    ▼         ▼         ▼
 agent_node  agent_node  planning_pipeline_node
    │ ⇄       │ ⇄       (1) 5类并行POI搜索
 tools_node  tools_node  (2) 语义去重
    │         │          (3) 多因子评分
    ▼         ▼          (4) 时长估算+动态容量
   END       END         (5) DBSCAN聚类分天
                         (6) 湖/山障碍检测
                         (7) 链式日间排序
                         (8) 跨日POI重分配
                         (9) 并行每日路线（信号量=3）
                         (10) 提交验证
                              │
                              ▼
                        summary_node (LLM摘要)
                              │
                              ▼
                            END
```

**关键设计决策：**
- 规划流水线不使用LLM逐步推理，而是确定性算法 —— 结果可复现、质量稳定
- 并行搜索用5类硬编码关键词覆盖95%的搜索需求，省去LLM策略生成（节省约3秒）
- Critic验证步骤被跳过（注释说明"节省约15秒，边际价值有限"）
- 高德API并发限制为3（信号量控制），避免超QPS

### 4.2 SSE流式协议

请求：`POST /api/v1/agent/chat`，Content-Type: `application/json`

```json
{"message": "帮我规划杭州三日游", "session_id": "uuid-xxx"}
```

响应：Content-Type: `text/event-stream`

所有事件格式：`event: message\ndata: {"type":"<type>","data":{...}}\n\n`

流结束标记：`event: done\ndata: {}\n\n`

20种事件类型：thinking, text, done, error, progress, tool_calling, poi_result, route_result, plan_summary, intent_detected, searching, candidates_ready, scoring, clustering, day_routing, critic_review, critic_result, routing, day_routed, validating

### 4.3 外部接口

| 服务 | 接口 | 调用位置 |
|------|------|---------|
| 高德POI搜索 | /v3/place/text | search_pois工具 |
| 高德周边搜索 | /v3/place/around | search_nearby工具 |
| 高德地理编码 | /v3/geocode/geo | geocode工具 |
| 高德路径规划 | /v3/direction/driving | plan_route工具 + route_one_day |
| LLM API | /v1/chat/completions (OpenAI兼容) | agent_node + classify_intent + summary |

### 4.4 模块间接口

**前端↔后端：** HTTP REST (JSON) + SSE (EventStream)，所有请求通过Vite代理转发（`/api` → `localhost:8000`）

**后端↔Agent：** 函数调用，后端创建LangGraph图实例，通过`astream_events`获取事件流，转化为SSE消息

**Agent↔工具：** LangChain StructuredTool接口，通过ToolNode注册，Planner通过LLM tool_call机制调用

## 5 数据流设计

### 5.1 主流程数据流

```
用户输入 "杭州三日游"
    │
    ▼
classify_intent_node ──► intent=trip_planning, city=杭州, days=3
    │
    ▼
planning_pipeline_node
    │
    ├─(1) 并行搜索: 5类×Amap → candidate_pois = [42个]
    ├─(2) 去重: deduplicate_pois → 40个
    ├─(3) 评分: score_pois → 40个(score排序)
    ├─(4) 时长: estimate_duration → 动态 max_per_day=6
    ├─(5) 分区: geo_partition → [day1: 6poi, day2: 7poi, day3: 6poi]
    ├─(6) 障碍: Amap驾车距离检测 → 跨湖拆分
    ├─(7) 排序: 链式排序确保Day1终点≈Day2起点
    ├─(8) 交换: 边界POI跨日重分配
    ├─(9) 路线: route_one_day×3(并行,信号量3) → polylines+segments
    ├─(10) 提交: submit_plan → DB持久化
    │
    ▼
summary_node → LLM生成自然语言摘要
    │
    ▼
SSE事件流 → 前端更新对话+地图
```

### 5.2 Agent状态数据结构

```python
class AgentState(TypedDict):
    messages: list[BaseMessage]    # 对话历史
    session_id: str                # 会话UUID
    tool_call_count: int           # 工具调用计数（上限25）
    error: str | None              # 错误信息
    intent: str                    # chat|poi_query|route_query|trip_planning
    intent_city: str | None        # 目标城市
    intent_days: int | None        # 旅行天数
    intent_preferences: list[str]  # 偏好标签
    plan_result: dict | None       # 规划结果
    plan_city: str                 # 规划城市
    plan_days: int                 # 规划天数
```
