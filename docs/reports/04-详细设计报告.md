## 1 引言

### 1.1 编写目的

本文档是"基于AI Agent的旅游路线智能规划与景点空间推荐系统"课程设计项目的详细设计说明书，旨在对系统各模块的内部实现逻辑、数据结构、算法流程及接口协议进行精确描述，为后续编码实现、测试验证及系统维护提供技术依据。

- **测试人员**：依据接口定义与算法逻辑编写测试用例。

### 1.2 项目背景

随着旅游产业的数字化转型，游客对个性化、智能化的出行规划需求日益增长。传统旅游推荐系统多采用静态规则引擎或简单的协同过滤算法，难以应对多约束、多目标的复杂规划场景。

本项目以大语言模型（LLM）驱动的智能Agent为核心决策层，结合PostGIS空间分析能力与多因子推荐引擎，实现"自然语言交互 → 空间智能分析 → 路线自动优化"的端到端规划链路。系统采用LangGraph StateGraph编排Agent工作流，通过SSE（Server-Sent Events）协议向前端推送结构化结果与流式文本，在前端以Leaflet地图实时渲染路线与标注点。

## 2 Agent详细设计

### 2.1 StateGraph结构定义

系统采用LangGraph的StateGraph作为Agent工作流的编排框架。StateGraph是一种有向图结构，节点（Node）代表处理逻辑单元，边（Edge）代表状态转移路径。

#### 2.1.1 状态图总览

```
                    ┌─────────────┐
                    │   START     │
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │   Router    │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
    ┌─────────▼──┐  ┌─────▼──────┐ ┌──▼──────────┐
    │  Planner   │  │  General   │ │  Clarifier  │
    │   Agent    │  │  Response  │ │  (追问补全)  │
    └─────────┬──┘  └─────┬──────┘ └──┬──────────┘
              │            │            │
              └────────────┼────────────┘
                           │
                    ┌──────▼──────┐
                    │  Formatter  │
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │    END      │
                    └─────────────┘
```

**图2-1 Agent StateGraph状态转移图**

#### 2.1.2 节点定义

| 节点名称 | 类型 | 职责 |
|----------|------|------|
| Router | 条件路由节点 | 调用LLM进行意图分类，根据分类结果选择下游节点 |
| Planner Agent | ReAct推理节点 | 核心决策大脑，通过ReAct循环自主调用工具完成规划 |
| General Response | 直接响应节点 | 处理非规划类的一般性对话 |
| Clarifier | 追问补全节点 | 当必要参数缺失时，生成追问消息引导用户补充 |
| Formatter | 格式化输出节点 | 将结构化数据与自然语言组装为SSE事件流 |

#### 2.1.3 边定义

```python
# 条件路由函数
def router_route(state: AgentState) -> str:
    """根据Router节点的意图分类结果决定下游节点"""
    intent = state.get("intent")
    if intent == "trip_planning" or intent == "poi_recommendation":
        if not state.get("city") or not state.get("days"):
            return "clarifier"
        return "planner_agent"
    else:
        return "general_response"

# 图的构建
graph = StateGraph(AgentState)
graph.add_node("router", router_node)
graph.add_node("planner_agent", planner_node)
graph.add_node("general_response", general_response_node)
graph.add_node("clarifier", clarifier_node)
graph.add_node("formatter", formatter_node)

# 边连接
graph.add_edge(START, "router")
graph.add_conditional_edges("router", router_route, {
    "planner_agent": "planner_agent",
    "general_response": "general_response",
    "clarifier": "clarifier"
})
graph.add_edge("planner_agent", "formatter")
graph.add_edge("general_response", "formatter")
graph.add_edge("clarifier", "formatter")
graph.add_edge("formatter", END)
```

### 2.2 AgentState数据结构

AgentState是贯穿整个StateGraph的全局状态对象，采用TypedDict定义，各节点通过读写该状态实现数据传递。

```python
class AgentState(TypedDict):
    messages: list[BaseMessage]        # 完整对话历史（含system/user/assistant/tool消息）
    session_id: str                    # 会话唯一标识符，用于关联同一轮交互
    intent: str                        # 意图分类结果: trip_planning | poi_recommendation | general
    city: str | None                   # 目标城市名称（由LLM从对话中提取）
    days: int | None                   # 旅行天数
    preferences: list[str]             # 用户偏好标签列表，如["历史", "美食", "自然"]
    companion_types: list[str]         # 同行人员类型: ["children", "elderly", "solo", "couple"]
    budget_level: int | None           # 预算等级: 1=经济 2=舒适 3=豪华
    candidate_pois: list[dict]         # 空间过滤后的候选景点集合
    selected_pois: list[dict]          # 推荐引擎筛选后的最终推荐景点
    daily_plans: list[dict]            # 按天组织的行程计划
    route_polylines: list[str]         # 路线编码数据（GeoJSON/Encoded Polyline）
    recommendation_weights: dict       # Agent根据用户画像动态配置的推荐权重
    response_text: str                 # 最终的自然语言响应文本
    structured_plan: dict | None       # 结构化行程数据（供前端渲染使用）
```

**字段详细说明**：

| 字段 | 写入节点 | 读取节点 | 数据类型 | 说明 |
|------|---------|---------|---------|------|
| messages | 全链路 | 全链路 | list[BaseMessage] | LangChain消息对象列表，包含对话完整上下文 |
| intent | Router | Planner, Formatter | str | 枚举值，决定路由走向 |
| recommendation_weights | Planner | 推荐引擎 | dict | 五维权重向量，格式为{pref, dist, rating, time, pop} |
| candidate_pois | Planner（工具调用） | 推荐引擎 | list[dict] | 每个dict包含poi_id, name, lng, lat, category, rating等 |
| daily_plans | Planner（工具调用） | Formatter | list[dict] | 每天包含date, pois[], route_info, duration等 |

### 2.3 Router节点设计

├────────────────┼─────────────┼───────────────────┤

**Router System Prompt**：

def router_node(state: AgentState) -> dict:
    """Router节点处理逻辑"""
    # 1. 构建分类prompt
    user_message = state["messages"][-1].content
    classification_prompt = build_classification_prompt(user_message)

    }

### 2.4 Planner Agent设计

#### 2.4.1 ReAct循环流程

Planner Agent采用ReAct（Reasoning and Acting）模式运行，即LLM在每一步先进行推理思考（Thought），然后决定调用某个工具（Action），获取工具返回结果（Observation）后继续下一轮推理，直至得出最终结果。

```
┌─────────────────────────────────────────────────┐
│                ReAct 循环                        │
│                                                  │
│   ┌──────────┐                                   │
│   │ Thought  │ ← LLM推理: 分析当前状态          │
│   └────┬─────┘                                   │
│        │                                         │
│   ┌────▼─────┐                                   │
│   │  Action  │ ← 决定调用工具或输出最终结果       │
│   └────┬─────┘                                   │
│        │                                         │
│   ┌────▼──────┐                                  │
│   │Observation│ ← 获取工具返回结果                │
│   └────┬──────┘                                  │
│        │                                         │
│        ▼                                         │
│   是否完成？──否──→ 回到 Thought                  │
│       │                                          │
│      是                                          │
│       │                                          │
│       ▼                                          │
│   输出最终结果                                    │
└─────────────────────────────────────────────────┘
```

#### 2.4.3 工具调用策略

Planner Agent的工具调用遵循以下策略：

1. **参数注入**：Router节点提取的city、days等参数通过State传递给Planner，减少Planner的推理负担；
2. **级联调用**：先调用search_pois获取候选集，再调用spatial_cluster进行聚类分天，最后调用optimize_tsp_order优化每日路线；
3. **最大迭代限制**：ReAct循环最多执行10轮，超出后强制输出当前最优结果；
4. **错误恢复**：工具调用失败时，Agent可尝试替代工具或降低要求重新调用。

### 2.5 Formatter节点设计

Formatter节点负责将Planner Agent输出的结构化数据转换为前端可消费的SSE事件序列。

#### 2.5.1 SSE消息组装逻辑

```python
async def formatter_node(state: AgentState) -> AsyncIterator[str]:
    """Formatter节点：将结构化数据组装为SSE事件流"""
    
    # 1. 发送思考过程（如果有）
    if state.get("thinking_text"):
        yield format_sse("thinking", state["thinking_text"])
    
    # 2. 发送POI标注数据
    if state.get("selected_pois"):
        poi_data = {
            "pois": state["selected_pois"],
            "center": calculate_center(state["selected_pois"]),
            "zoom": calculate_zoom_level(state["selected_pois"])
        }
        yield format_sse("poi_result", poi_data)
    
    # 3. 发送路线数据
    if state.get("route_polylines"):
        route_data = {
            "polylines": state["route_polylines"],
            "daily_plans": state["daily_plans"]
        }
        yield format_sse("route_result", route_data)
    
    # 4. 发送行程摘要卡片
    if state.get("structured_plan"):
        yield format_sse("plan_summary", state["structured_plan"])
    
    # 5. 流式输出自然语言文本
    async for chunk in stream_text(state["response_text"]):
        yield format_sse("text", chunk)
    
    # 6. 发送结束信号
    yield "event: done\ndata: {}\n\n"
```

#### 2.5.2 SSE消息格式规范

每条SSE消息遵循以下格式：

```
event: message\ndata: {"type": "<消息类型>", "content": <内容>}\n\n
```

| 消息类型 | content类型 | 触发条件 | 前端行为 |
|---------|-----------|---------|---------|
| thinking | string | Agent开始推理 | 显示"正在思考..."动画 |
| tool_calling | string | Agent调用工具 | 显示工具调用状态 |
| poi_result | object | POI推荐完成 | 地图上标注POI点 |
| route_result | object | 路线规划完成 | 地图上绘制路线 |
| plan_summary | object | 行程摘要生成 | 显示行程卡片 |
| text | string | 文本流式输出 | 逐字显示文字 |
| error | object | 发生错误 | 显示错误提示 |

def _build_fallback_prompt(self, messages, tools):
    """构建降级模式下的prompt"""
    tool_descriptions = "\n".join([
        f"- {t.name}: {t.description}\n  参数: {json.dumps(t.parameters)}"
        for t in tools
    ])
    
    system_msg = f"""你可以调用以下工具来完成任务：
{tool_descriptions}

当你需要调用工具时，请按以下JSON格式输出：
{{"tool": "工具名称", "arguments": {{参数}}}}

当你不需要调用工具时，直接输出回复文本。"""
    
    return [SystemMessage(content=system_msg)] + messages

## 3 推荐引擎详细设计

### 3.1 整体流水线设计

推荐引擎采用五步流水线架构，从原始POI数据库到最终推荐结果依次经过：空间过滤 → 多因子评分 → MMR多样性重排 → DBSCAN空间聚类分天 → TSP路线优化。

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 空间过滤  │───→│ 多因子评分│───→│ MMR重排  │───→│ 聚类分天  │───→│ TSP优化  │
│ PostGIS  │    │ Multi-   │    │ Maximal  │    │ DBSCAN   │    │ 最近邻+  │
│ ~50候选   │    │ Factor   │    │ Marginal │    │ 分天分组  │    │ 2-opt    │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
   输入            候选集          评分排序集       分天计划        最终路线
   全量POI        ~50个           Top-K个          K个/天          优化顺序
```

### 3.2 空间过滤算法

空间过滤利用PostGIS的空间索引（GiST）快速筛选目标区域内的候选POI，将搜索范围从全库缩小至合理规模。

**核心SQL查询**：

```sql
-- 以用户指定城市中心点为基准，查询指定半径内的POI
SELECT 
    p.poi_id,
    p.name,
    p.category,
    p.rating,
    p.longitude,
    p.latitude,
    ST_Distance(
        p.geom::geography, 
        ST_SetSRID(ST_MakePoint(:center_lng, :center_lat), 4326)::geography
    ) AS distance
FROM pois p
WHERE ST_DWithin(
    p.geom::geography,
    ST_SetSRID(ST_MakePoint(:center_lng, :center_lat), 4326)::geography,
    :radius_meters
)
AND p.category IN (:categories)
AND p.rating >= :min_rating
ORDER BY distance ASC
LIMIT :limit;
```

**参数配置**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| radius_meters | 15000 | 搜索半径（米），城市级推荐默认15km |
| min_rating | 3.5 | 最低评分阈值 |
| limit | 50 | 候选集上限 |

### 3.3 多因子评分算法

#### 3.3.1 评分公式

对每个候选POI $p_i$，综合评分计算公式为：

$$S(p_i) = w_1 \cdot F_{pref}(p_i) + w_2 \cdot F_{dist}(p_i) + w_3 \cdot F_{rating}(p_i) + w_4 \cdot F_{time}(p_i) + w_5 \cdot F_{pop}(p_i)$$

其中权重向量 $\vec{w} = (w_1, w_2, w_3, w_4, w_5)$ 满足 $\sum_{j=1}^{5} w_j = 1$，由Agent根据用户画像动态配置。

#### 3.3.2 各因子计算方法

**偏好匹配因子 $F_{pref}$**：

$$F_{pref}(p_i) = \frac{|Tags(p_i) \cap Preferences_{user}|}{|Preferences_{user}|}$$

其中 $Tags(p_i)$ 为POI的标签集合，$Preferences_{user}$ 为用户偏好标签集合。

**空间距离因子 $F_{dist}$**：

$$F_{dist}(p_i) = 1 - \frac{dist(p_i, center)}{radius_{max}}$$

距离越近得分越高，归一化到[0, 1]区间。

**评分因子 $F_{rating}$**：

$$F_{rating}(p_i) = \frac{rating(p_i)}{5.0}$$

直接将POI评分归一化。

**时间适宜性因子 $F_{time}$**：

$$F_{time}(p_i) = \begin{cases} 1.0 & \text{if 当前季节适合该POI} \\ 0.5 & \text{if 一般适合} \\ 0.2 & \text{if 不适合（如冬季推荐水上乐园）} \end{cases}$$

**人气因子 $F_{pop}$**：

$$F_{pop}(p_i) = \frac{review\_count(p_i)}{\max_{j}(review\_count(p_j))}$$

以候选集中评论数最大值归一化。

#### 3.3.3 权重配置策略

Agent根据用户画像动态调整权重向量，预定义策略模板如下：

| 用户类型 | $w_{pref}$ | $w_{dist}$ | $w_{rating}$ | $w_{time}$ | $w_{pop}$ |
|---------|-----------|-----------|-------------|-----------|----------|
| 家庭亲子游 | 0.25 | 0.15 | 0.25 | 0.25 | 0.10 |
| 背包客/自由行 | 0.35 | 0.30 | 0.10 | 0.10 | 0.15 |
| 老年团游 | 0.15 | 0.20 | 0.30 | 0.25 | 0.10 |
| 情侣游 | 0.30 | 0.10 | 0.20 | 0.20 | 0.20 |
| 默认 | 0.20 | 0.20 | 0.20 | 0.20 | 0.20 |

### 3.4 MMR多样性重排算法

MMR（Maximal Marginal Relevance）算法用于在保持推荐质量的同时增加结果多样性，避免推荐同质化景点。

**算法伪代码**：

```
function MMR_Rerank(candidates, k, λ):
    Input: candidates — 候选POI集合
           k — 需要选择的数量
           λ — 平衡参数（0.7）
    
    Selected = empty_set
    Remaining = candidates
    
    while |Selected| < k and Remaining is not empty:
        best_score = -∞
        best_poi = null
        
        for each poi in Remaining:
            # 相关性得分（多因子评分）
            relevance = score(poi)
            
            # 与已选集合的最大相似度
            max_sim = max(similarity(poi, s) for s in Selected)
            if Selected is empty:
                max_sim = 0
            
            # MMR得分
            mmr_score = λ * relevance - (1 - λ) * max_sim
            
            if mmr_score > best_score:
                best_score = mmr_score
                best_poi = poi
        
        Selected.add(best_poi)
        Remaining.remove(best_poi)
    
    return Selected
```

**相似度定义**：

$$sim(p_i, p_j) = \alpha \cdot category\_overlap(p_i, p_j) + (1 - \alpha) \cdot proximity(p_i, p_j)$$

其中 $\alpha = 0.6$，$proximity$ 为空间距离的倒数归一化值。

### 3.5 DBSCAN空间聚类算法

DBSCAN（Density-Based Spatial Clustering of Applications with Noise）用于将推荐景点按空间分布聚类分组，每天的游览景点归入同一聚类。

**参数配置**：

| 参数 | 值 | 说明 |
|------|---|------|
| eps | 3000m | 邻域半径，同一聚类内景点间最大距离 |
| min_samples | 2 | 核心点最少邻域样本数 |
| metric | haversine | 球面距离度量 |

**算法伪代码**：

```
function DBSCAN_Cluster(pois, eps, min_samples):
    Input: pois — 景点集合（含经纬度）
           eps — 邻域半径（米）
           min_samples — 最小样本数
    
    clusters = []
    visited = empty_set
    
    for each poi in pois:
        if poi in visited:
            continue
        
        neighbors = range_query(poi, eps)
        
        if |neighbors| < min_samples:
            # 标记为噪声点，后续单独处理
            mark poi as noise
            continue
        
        # 新建聚类，扩展密度可达点
        cluster = new_cluster()
        expand_cluster(poi, neighbors, cluster, visited, eps, min_samples)
        clusters.add(cluster)
    
    return clusters

function expand_cluster(poi, neighbors, cluster, visited, eps, min_samples):
    visited.add(poi)
    cluster.add(poi)
    
    queue = neighbors
    while queue is not empty:
        current = queue.dequeue()
        if current not in visited:
            visited.add(current)
            current_neighbors = range_query(current, eps)
            if |current_neighbors| >= min_samples:
                queue.enqueue_all(current_neighbors)
            cluster.add(current)
```

**聚类后分天策略**：按聚类大小排序，每天分配1个聚类，小聚类合并到相邻大聚类，噪声点分配到最近的聚类。

### 3.6 TSP路线优化算法

#### 3.6.1 最近邻启发式初始解

```
function Nearest_Neighbor(pois, start):
    Input: pois — 当天景点集合
           start — 起始景点（距用户最近的景点或酒店）
    
    route = [start]
    unvisited = pois - {start}
    
    while unvisited is not empty:
        current = route.last()
        nearest = argmin(distance(current, p) for p in unvisited)
        route.append(nearest)
        unvisited.remove(nearest)
    
    return route
```

#### 3.6.2 2-opt改进算法

```
function Two_Opt_Improve(route, max_iterations):
    Input: route — 初始路线
           max_iterations — 最大迭代次数
    
    improved = true
    iterations = 0
    
    while improved and iterations < max_iterations:
        improved = false
        iterations += 1
        
        for i = 1 to |route| - 2:
            for j = i + 1 to |route| - 1:
                # 计算交换前后距离变化
                d_old = dist(route[i-1], route[i]) + dist(route[j], route[j+1])
                d_new = dist(route[i-1], route[j]) + dist(route[i], route[j+1])
                
                if d_new < d_old:
                    # 反转route[i..j]段
                    route[i..j] = reverse(route[i..j])
                    improved = true
    
    return route
```

**复杂度分析**：2-opt单次迭代时间复杂度为 $O(n^2)$，其中 $n$ 为当天景点数量（通常3-5个），实际计算量极小，可在毫秒级完成。

    const response = await fetch('/api/v1/agent/chat', {
        method: 'POST',
        headers: {
            'Content-Type': 'application/json',
            'Authorization': `Bearer ${token}`
        },
        body: JSON.stringify({ session_id: sessionId, message })
    });

    const reader = response.body!.getReader();
    const decoder = new TextDecoder();

    while (true) {
        const { done, value } = await reader.read();
        if (done) break;

        const text = decoder.decode(value);
        const events = parseSSEEvents(text);

        for (const event of events) {
            switch (event.type) {
                case 'poi_result':
                    mapStore.updatePois(event.content.pois);
                    break;
                case 'route_result':
                    mapStore.updateRoutes(event.content.polylines);
                    break;

                    break;
                case 'text':
                    appendToLastMessage(event.content);
                    break;
                case 'error':
                    showError(event.content.message);
                    break;
            }
        }
    }

### 5.4 状态管理（Pinia Store设计）

系统采用Pinia作为前端状态管理方案，分为三个Store模块：

```typescript
// stores/chat.ts — 对话状态
interface ChatStore {
    messages: Message[];           // 消息列表
    isLoading: boolean;            // 是否正在等待Agent响应
    sessionId: string;             // 当前会话ID
    // Actions
    sendMessage: (msg: string) => Promise<void>;
    appendStreamText: (text: string) => void;
    addMessage: (msg: Message) => void;
}

// stores/map.ts — 地图状态
interface MapStore {
    center: [number, number];      // 地图中心
    zoom: number;                  // 缩放级别
    pois: PoiItem[];               // 当前POI列表
    routes: RouteData[];           // 路线数据
    selectedPoiId: string | null;  // 当前选中的POI
    // Actions
    updatePois: (pois: PoiItem[]) => void;
    updateRoutes: (routes: RouteData[]) => void;
    flyToPoi: (poiId: string) => void;
    fitAllPois: () => void;
}

// stores/trip.ts — 行程状态
interface TripStore {
    currentPlan: PlanSummary | null;  // 当前规划方案
    savedTrips: Trip[];               // 已保存行程列表
    // Actions
    saveTrip: (plan: PlanSummary) => Promise<void>;
    loadTrips: () => Promise<void>;
    deleteTrip: (tripId: string) => Promise<void>;
}
```

**Store交互关系**：

```
  ChatStore ──(poi_result)──→ MapStore.updatePois()
      │                           │
      │(plan_summary)              │(selectedPoi)
      ▼                           ▼
  TripStore                   TimelineView
      │                           │
      │(load/save)                 │(click poi)
      ▼                           ▼
   API Server                MapStore.flyToPoi()
```

## 6 工具链详细设计

系统共注册14个工具供Planner Agent调用。以下逐一描述各工具的输入输出规格、处理逻辑及错误处理策略。

### 6.1 search_pois

**功能**：按城市、关键词、类别搜索POI。

| 项目 | 说明 |
|------|------|
| **输入** | `city: str`（必填）, `keyword: str`（可选）, `category: str`（可选）, `limit: int`（默认20） |
| **输出** | `list[dict]`，每个dict含 `poi_id, name, lng, lat, category, rating, address, tags` |
| **处理逻辑** | 1. 调用geocode获取城市中心坐标<br>2. 构造PostGIS空间查询<br>3. 按评分和距离综合排序<br>4. 返回Top-N结果 |
| **错误处理** | 城市未找到 → 返回空列表+错误信息；数据库连接失败 → 抛出异常由Agent捕获 |

### 6.2 search_nearby

**功能**：搜索指定坐标附近的POI。

| 项目 | 说明 |
|------|------|
| **输入** | `lng: float`, `lat: float`, `radius: int`（米，默认3000）, `category: str`（可选） |
| **输出** | `list[dict]`，格式同search_pois |
| **处理逻辑** | PostGIS ST_DWithin查询，按距离升序排列 |
| **错误处理** | 坐标越界 → 参数校验错误 |

### 6.3 geocode

**功能**：地址文本编码为经纬度坐标。

| 项目 | 说明 |
|------|------|
| **输入** | `address: str`（如"杭州市西湖景区"） |
| **输出** | `{"lng": float, "lat": float, "formatted_address": str}` |
| **处理逻辑** | 调用地理编码服务（高德/OSM Nominatim），解析返回结果 |
| **错误处理** | 地址无法识别 → 返回null + 错误提示 |

### 6.4 reverse_geocode

**功能**：经纬度坐标逆编码为地址。

| 项目 | 说明 |
|------|------|
| **输入** | `lng: float`, `lat: float` |
| **输出** | `{"address": str, "components": dict}` |
| **处理逻辑** | 调用逆地理编码服务 |
| **错误处理** | 坐标无对应地址 → 返回最近匹配地址 |

### 6.5 plan_route

**功能**：规划两点或多点之间的路线。

| 项目 | 说明 |
|------|------|
| **输入** | `waypoints: list[{lng, lat}]`（途经点列表）, `mode: str`（"driving"/"walking"/"transit"/"cycling"） |
| **输出** | `{"distance": float（米）, "duration": int（秒）, "polyline": str（编码坐标串）, "steps": list[dict]}` |
| **处理逻辑** | 调用路径规划API（OSRM/高德），解析路线几何数据 |
| **错误处理** | 路径不可达 → 返回直线距离估算 + 警告标记 |

### 6.6 get_poi_detail

**功能**：获取单个POI的详细信息。

| 项目 | 说明 |
|------|------|
| **输入** | `poi_id: str` |
| **输出** | `dict`含完整信息（名称、地址、开放时间、门票价格、电话、图片URL、评论摘要等） |
| **处理逻辑** | 数据库查询POI详情表 |
| **错误处理** | POI不存在 → 返回null + 错误信息 |

### 6.7 get_weather

**功能**：查询城市天气预报。

| 项目 | 说明 |
|------|------|
| **输入** | `city: str`, `date_range: [str, str]`（起止日期） |
| **输出** | `list[dict]`，每天含 `date, temp_high, temp_low, condition, precipitation_prob` |
| **处理逻辑** | 调用天气API（和风天气/OpenWeatherMap），支持未来7天预报 |
| **错误处理** | 城市不支持 → 返回空列表 + 降级提示 |

### 6.8 query_db_pois

**功能**：通过空间范围（bbox）直接查询POI数据库。

| 项目 | 说明 |
|------|------|
| **输入** | `bbox: [min_lng, min_lat, max_lng, max_lat]`, `category: str`（可选）, `min_rating: float`（默认0） |
| **输出** | `list[dict]`，同search_pois格式 |
| **处理逻辑** | PostGIS ST_Within查询，利用GiST空间索引加速 |
| **错误处理** | bbox无效 → 参数校验错误 |

### 6.9 spatial_cluster

**功能**：对POI集合进行DBSCAN空间聚类。

| 项目 | 说明 |
|------|------|
| **输入** | `pois: list[dict]`（含lng, lat）, `eps: float`（默认3000米）, `min_samples: int`（默认2） |
| **输出** | `{"clusters": list[list[dict]], "noise": list[dict]}` |
| **处理逻辑** | 实现DBSCAN算法（见3.5节），使用haversine距离度量 |
| **错误处理** | POI数量不足 → 返回单聚类；坐标异常 → 过滤异常点 |

### 6.10 calculate_route_score

**功能**：计算一组POI的路线综合评分。

| 项目 | 说明 |
|------|------|
| **输入** | `pois: list[dict]`, `weights: dict`（五维权重） |
| **输出** | `{"total_score": float, "poi_scores": list[{poi_id, score, factors}]}` |
| **处理逻辑** | 实现多因子评分算法（见3.3节），逐POI计算后加权汇总 |
| **错误处理** | 权重之和不等于1 → 自动归一化并警告 |

### 6.11 optimize_tsp_order

**功能**：优化POI的游览顺序（TSP求解）。

| 项目 | 说明 |
|------|------|
| **输入** | `pois: list[dict]`（含lng, lat） |
| **输出** | `{"ordered_pois": list[dict], "total_distance": float, "route": list[{from, to, distance}]}` |
| **处理逻辑** | 最近邻启发式 + 2-opt局部优化（见3.6节） |
| **错误处理** | POI数量≤2 → 直接返回；>20 → 分段优化 |

### 6.12 get_travel_tips

**功能**：获取城市旅行建议。

| 项目 | 说明 |
|------|------|
| **输入** | `city: str`, `season: str`（"spring"/"summer"/"autumn"/"winter"） |
| **输出** | `{"tips": list[str], "best_months": list[str], "warnings": list[str]}` |
| **处理逻辑** | 查询预置知识库或调用LLM生成 |
| **错误处理** | 城市无数据 → 返回通用旅行建议 |

### 6.13 check_poi_hours

**功能**：查询POI在指定日期时间的开放状态。

| 项目 | 说明 |
|------|------|
| **输入** | `poi_id: str`, `date_time: str`（ISO格式） |
| **输出** | `{"is_open": bool, "hours": str, "note": str}` |
| **处理逻辑** | 查询POI开放时间规则表，处理节假日特殊情况 |
| **错误处理** | 无时间数据 → 返回`is_open: null, note: "时间信息暂缺"` |

### 6.14 estimate_visit_duration

**功能**：估算POI的游览时长。

| 项目 | 说明 |
|------|------|
| **输入** | `poi_id: str` |
| **输出** | `{"duration_minutes": int, "suggested_time": str（如"2-3小时"）, "factors": list[str]}` |
| **处理逻辑** | 基于POI类型、面积、历史游览数据估算<br>规则：大型景区3-5h，博物馆2-3h，小型景点1-2h |
| **错误处理** | 无历史数据 → 使用类型默认值 |
